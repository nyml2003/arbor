# SOP：从一个项目抽取设计经验

## 适用场景

你发现了一个有架构价值的项目（自己写的或别人的），想把设计模式抽出来放入 learn/patterns/。

## 流程

### Step 1：判断值不值得抽

先看三个维度，确认值得抽再动手：

- 有没有独特的架构设计？（不是脚手架模板、不是 framework tutorial 级别）
- 有没有跨项目的模式？（可能和已有的 pattern 共鸣）
- 是不是纯文档/空壳？（纯文档跳过，等有实现再抽）

如果三个问题的答案都是"没有"，跳过。不值得花时间。

### Step 2：读代码

**必须读源码，不能只读 README。** README 告诉你它做什么，源码告诉你它怎么做的。

- 找到核心模块（一般以 `src/` 或 `lib/` 为入口）
- 按依赖方向读：core → domain → application → infrastructure → cli/ui
- 关注接口/类型定义多于关注实现细节
- 读测试——测试告诉你边界条件和期望行为

读到什么程度算够了：你能用自己的话解释清楚它的核心架构是什么、为什么这么设计、有哪些关键的类型/接口/状态机。

### Step 3：识别模式

从代码中提取 3-6 个关键设计决策：

- 它用了什么抽象？（trait、interface、Protocol）
- 数据怎么流动？（top-down？事件驱动？pipeline？）
- 错误怎么处理？（异常？Result？error code？）
- 状态怎么管理？（状态机？事件溯源？无状态？）
- 层怎么分？（单向依赖？哪层依赖哪层？）
- 有什么明确的设计取舍？（做了什么、明确不做什么）

### Step 4：写文档

文件路径：`learn/patterns/<descriptive-slug>.md`

#### 文档模板

```markdown
# 模式：<主题>（<项目名>）

## 一句话

<用一句话解释这个项目的核心设计>

## 为什么需要这个模式

<这个模式解决什么问题，为什么通用——不引用任何特定项目>

## 核心架构

<文字描述 + ASCII 图展示结构>

## 关键设计

### 1. <设计点 1>

<解释 + 代码片段>

### 2. <设计点 2>

...

## 反模式警示

### ❌ <常见错误>

<解释为什么错>

## 来源

- <项目名> 源码（<路径>）
- <日期> agent 阅读后提炼
```

#### 硬约束

- **不引用 Arbor。** 不写"Arbor 的对应"、"对 Arbor 有用"、"可以在 Arbor 中复制"。这是独立文档。
- **不引用其他 pattern。** 每篇独立。相关性在 patterns/README.md 的索引里体现。
- **来源必须包含文件路径。** 让读者可以自己去看原始代码。
- **代码片段必须有上下文。** 不贴孤立一行。

### Step 5：更新索引

写完文档后，更新 `patterns/README.md`：

- 在"按来源项目"表加一行
- 在"按主题"表加对应的主题关联

### Step 6：记录

- 在 `iteration-log/` 加一条记录（做了什么、学到了什么、决策）
- 如果是旧项目抽取，在 `manage/tasks.md` 中标记该项目为 ✅

## 反模式

- ❌ 只看 README 不读源码 → 写出来的 pattern 是表面描述
- ❌ 用 Arbor 视角过滤 → 要描述项目自己的设计，不是"对 Arbor 有用的部分"
- ❌ 同时抽多个项目 → 一次做一个，做完更新索引，再做下一个
- ❌ 抽完不更新索引 → 一个月后找不到
