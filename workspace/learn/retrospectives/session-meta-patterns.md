# 复盘：全场对话的元模式

日期：2026-06-07

## 十五次用户纠正

回溯整个对话，用户纠正了 agent 的 15 个错误。按性质分类：

### 方向性错误（5 次）

| # | agent 做了什么 | 用户纠正 | 根因 |
|---|--------------|---------|------|
| 1 | 建议用 bun compile 打 CLI 二进制 | 别用 bun，参考 ObolosFS | 没看现有方案就选工具 |
| 2 | 建议引入 marked 库做 Markdown 渲染 | 为什么引入库，要自己实现 | 没对齐"自己造" |
| 3 | 被否后立刻"手写 lexer，现在开始" | 你又急了 | 换了实现方式，没换思维 |
| 4 | "你觉得呢"→ 建议做 Markdown 渲染 | 这个判断我不认可 | agent 的判断不代表用户判断 |
| 5 | 说"经验已内化，代码可删" | 经验还没内化呢 | agent 急于跳过沉淀步骤 |

**模式**：agent 的默认方向是"尽快出代码"。任何阻碍这个方向的信息（库不好用？自己写；自己的写来不及？引入库）都被 agent 自动选了一条——但两条都没对齐"自己造 + 不急于出码"这个大方向。

### 信息遗漏（4 次）

| # | agent 错过了什么 | 用户指出 |
|---|----------------|---------|
| 6 | work-context/repos/ 里 31 个项目——agent 只看 Python 代码，漏了 repos | "repos里有大量项目，你好像忽略了" |
| 7 | 忽略了项目的功能价值和参考价值，只看经验价值 | "除了按经验分，也按功能分" |
| 8 | 忽略了 docs/ 和 workspace/ 功能重合 | "docs和workspace的功能是不是重合了" |
| 9 | 没发现 learn/ 已经开始腐化 | "learn开始腐化了，根本不知道怎么看" |

**模式**：agent 倾向于只看一层——看到了经验价值就忽略了功能价值，看到了代码就忽略了 repos 子目录。需要多维度审视，需要有人不断提醒"还有呢？"。

### 视角偏差（3 次）

| # | agent 的视角 | 用户的视角 |
|---|------------|----------|
| 10 | Pattern 应该标注"对 Arbor 有什么价值" | "不可以有，这个应该在arbor本身的演进文档里" |
| 11 | 用 Arbor 视角筛选项目经验 | 项目经验应该独立描述，有耦合就歪了 |
| 12 | 判断 demo 没有价值 | "demo应该也挺最佳实践的" |

**模式**：agent 容易用当前项目的视角去看一切——"这个对 Arbor 有用吗"成为了唯一的评判标准。但好的知识库应该是多中心的——每个项目有自己独立的价值。

### 架构敏感性差（3 次）

| # | agent 的行为 | 用户指出 |
|---|------------|---------|
| 13 | MIGRATION.md 放根目录 | 工作文档不该在根目录 |
| 14 | sops/ 里放了技术操作手册 | SOP 应该是四环节流程 |
| 15 | manage/.gitkeep 残留 | 有了内容就应该删掉 |

**模式**：agent 对"文件放在哪里"的敏感度不够。文件位置本身就是架构信息——放错位置比不放更糟糕。

## 发现的深层模式

### 1. build 是最危险的分支

agent 反复试图跳到 build（写代码）。build 给人"有进展"的错觉，但 Arbor 真正的瓶颈是方向收敛和经验沉淀——这两个都在 learn 和 manage 里，不是代码能解决的。

### 2. 用户纠正的密度是信号

15 次纠正密集发生在一个会话里。这说明两件事：
- agent 需要更频繁地停下来确认方向，而不是假设方向是正确的
- 高密度纠正 = 需要写 retrospectives——纠正本身是最高价值的知识

### 3. 多维度评估是信息整理的基本功

看项目要看三个维度：经验（有什么可以学习）、功能（代码能不能复用）、参考（demo 是不是最佳实践）。只用一个维度看东西，一定漏信息。

### 4. 文件位置 = 架构信息

根目录放永久参考，分支里放工作文档，README 是目录的自我介绍。文件放错位置会造成信息断层——读者不知道该去哪找什么。这不是"整理"，是架构设计的一部分。

### 5. 正确的做事顺序

```
1. 对齐方向     ← build 之前先确认 build 什么
2. 沉淀认知     ← 删之前先抽经验
3. 发文决策     ← 写下来再做事
4. 执行         ← 最后才是写代码或删文件
```

agent 的默认顺序是反的：执行 → 发文决策 → 沉淀认知 → （从不）对齐方向。

## 已建立的防退化机制

| 机制 | 在哪里 | 作用 |
|------|--------|------|
| 不急于出码 | `retrospectives/agent-rush-to-code.md` | 每次想写代码时回头读 |
| 结构审视 | `retrospectives/structure-review-20260607.md` | 定期检查文件位置是否正确 |
| Pattern 独立 | patterns/ 全部脱耦 | 不用单一视角污染知识 |
| 三维度评估 | `manage/repos-inventory.md` | 以后看新项目用三个维度 |
| STATUS 状态 | `learn/README.md` | 总览当前进度，不被过程信息淹没 |
