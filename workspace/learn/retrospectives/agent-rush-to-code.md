# 复盘：agent 的"急于出码"惯性

日期：2026-06-07

## 现象

agent 在这轮交互中反复犯同一个错误：**遇到问题 → 立刻提技术方案 → 想写代码。** 被纠正了四次，模式一样。

## 四次纠正

| # | agent 做的 | 你说的 | 根本问题 |
|---|-----------|--------|---------|
| 1 | Phase 2 计划用 bun compile 打 CLI 二进制 | 别用 bun，参考 ObolosFS 的方案 | agent 没调研现有方案就直接选了工具 |
| 2 | "经验已内化，代码可删" | 经验还没内化呢 | agent 急于清项目，跳过沉淀步骤 |
| 3 | Markdown 渲染引入 marked/remark | 为什么引入库，要自己实现 | agent 没对齐 Arbor 的"自己造"目标 |
| 4 | 被否了立刻转向"手写 lexer，现在开始" | 你又急了 | 同上——从"引入库"变成"立即写码"，只是换了个实现方式，没换思维 |

## 根因

**agent 在拿"做事"（技术决策、写代码、删文件）替代"想清楚"。**

写代码和做技术决策对 agent 来说是最容易的输出——有明确的文件路径、工具调用、可见的产出。但 Arbor 真正的瓶颈不是缺代码：
- 50 个项目散落 → 需要的是分类和决策，不是代码
- 经验锁在代码里 → 需要的是读、理解、写下来，不是代码
- 方向不确定 → 需要的是收敛和澄清，不是代码

代码解决不了这三个问题。

## 正确节奏

```
1. 先对齐目标     ← 这个问题是 build 能解决的吗？
2. 再沉淀认知     ← 这个决策值得记下来吗？
3. 最后才写代码   ← 方向清了，写代码只需要一轮
```

### 具体行为准则

- 提技术方案之前，先问"这是不是对的方向"
- 说"可以引入 X 库"之前，先想"X 是我们自己想做的那类东西吗"
- 每个 Phase 做完，必须在 learn/ 下留记录
- 不急——Arbor 没有一个月就要上线的压力。方向对比速度快重要。

## 后果

不加纠正，agent 会把 Arbor 变成一个"用外部库快速堆起来的东西"——和"所有软件自己写"的大目标彻底矛盾。
