# 006 - Capture 文档和骨架先行

日期：2026-06-07

## 做了什么

1. 确定截图工具独立孵化，不挂进 Arbor 容器页面
2. 确定技术路线为 `Tauri 2 + Rust-first + 静态 overlay + 延迟加载的 SolidJS settings`
3. 在 `apps/capture/` 下建立文档和目录骨架
4. 写出产品说明、架构说明和测试规格
5. 在 `PLAN.md` 和 `DECISIONS.md` 里补上这条新路线

## 学到了什么

### 截图工具和 Arbor 容器不是一类东西

Arbor 容器是内容和工作流壳。截图工具是常驻系统工具。两者都叫“桌面应用”，但边界完全不同。前者可以接受 Electron，后者应该优先考虑更轻的宿主。

### 这次最该复用的是分层经验，不是运行时

Arbor 现有最有价值的部分不是 Electron 本身，而是这些工程经验：

- 先把边界写清楚
- 把副作用压到单独一层
- 让 UI 只负责 UI
- 先写文档和验收，再进入实现

### 先写骨架，能挡住后面的边界漂移

这次没有直接进 Tauri 初始化，也没有先写一个能跑的半成品。先把目录、职责、命令草案和测试规格写出来，后面实现时就不需要边写边想“这部分到底该放哪”。

## 决策

1. `apps/capture/` 独立建目录，不复用 `apps/container/`
2. 截图主链路全部下沉到 Rust
3. overlay 用静态页面和原生事件，settings 才用 SolidJS
4. 当前阶段只写文档和目录骨架，不写截图逻辑

## 下一步

- 进入实现阶段时，再补 Tauri 配置、Cargo 依赖和前端构建配置
- 优先打通区域截图主流程，不提前做标注和历史库
