# 模式：TypeScript UI 层性能规则

## 一句话

UI 层性能问题，先看“渲染了多少”和“什么时候渲染”，再看具体框架 API。最值得借鉴的做法有四个：按需挂载、细粒度更新、窗口化渲染、少做无意义布局。

## 这篇只讲什么

这篇只讲界面层能直接复用的规则：

- 长列表
- 树形结构
- 条件渲染
- 组件更新边界

不讲这些内容：

- runtime-core
- scheduler / parser / compiler
- TS 与 Rust 的边界判断

## 1. 不可见的内容不要先挂进 DOM

最简单也最有效的规则就是：用户现在看不到的东西，不要急着挂。

典型场景：

- 折叠树节点
- 未展开的详情区
- 条件分支
- 标签页里当前没打开的面板

比起“先渲染，再隐藏”，更稳的做法是条件挂载：

```tsx
<Show when={expanded()}>
  <TreeNodeList entries={children() ?? []} />
</Show>
```

这样做的好处：

- 少 DOM 节点
- 少事件绑定
- 少布局和绘制
- 少无意义的子树更新

如果一个节点根本不在 DOM 中，后面很多性能问题会一起消失。

## 2. 数据按需加载，不要一次把整棵树读出来

树形 UI 最容易犯的错，就是一上来把整棵结构都拉平或全量加载。

更稳的方式是：

- 顶层先读根节点
- 用户展开时再读子节点
- 已经读过的路径直接复用缓存

```tsx
const [children] = createResource(
  () => isExpanded() ? path : null,
  fetchDirectory
);
```

这条规则的本质不是某个框架 API，而是：

- 只为当前交互付费
- 把 IO 和渲染都限制在局部范围

如果以后节点数量继续涨，再考虑虚拟化。不要在几百个节点的树上提前做一套复杂长列表系统。

## 3. 长列表先窗口化，不要让 DOM 数量跟数据量线性增长

长列表的核心问题不是“滚动时卡”，而是 DOM 总量太大。

更稳的模型是：

- 只渲染可见窗口
- 上下放占位空白
- 滑动时移动窗口
- 必要时加少量 buffer

```ts
const start = Math.max(0, startIndex - bufferCount);
const end = Math.min(itemCount, startIndex + visibleCount + bufferCount);
const visibleItems = items.slice(start, end);
```

这个模型的价值很直接：

- 10000 条数据，不需要 10000 个 DOM
- 可见 10 项时，只渲染十几项就够

先问“能不能把 DOM 数量砍掉”，再问“某个节点还能不能更快一点”。

## 4. 先选细粒度更新模型，再考虑补丁技巧

UI 层很多性能问题，本质是更新粒度太粗。

粗粒度模型通常会这样：

- 父组件状态一变，整片子树都重新跑
- 只改一个字段，也要重过整段 render

更细的模型会把更新限制在真正变化的位置：

- 文本变了，只更新文本
- 一个节点展开，只挂这个节点的子树
- 列表窗口变了，只换可见窗口

如果框架本身支持细粒度响应，就顺着它的能力写。不要明明已经有局部更新能力，又在上层重新包成大颗粒状态。

## 5. 少做无意义布局，特别是滚动场景

滚动场景里，很多问题不是 JS 太慢，而是布局太频繁。

要优先避免这些写法：

- 高频改 `left` / `top`
- 一边读布局，一边写布局
- 为了隐藏内容保留一大堆绝对定位节点

更稳的写法通常是：

- 用正常文档流
- 用前后 spacer 占位
- 只改窗口范围

比如固定宽度横向虚拟滚动里，`flex + spacer` 往往比一排绝对定位更简单，也更不容易把布局搞乱。

## 6. 滚动和展开场景优先“简单但局部”的实现

很多 UI 性能问题不是因为实现太简单，而是因为实现太早变复杂。

例如文件树：

- 数量不大时，用递归 + 按需加载就够
- 不要一开始就上虚拟滚动

例如固定宽度列表：

- 宽度固定时，直接用窗口切片 + 占位
- 不要先上支持动态高度、滚动定位、复杂测量的通用库

性能优化不是抽象竞赛。简单实现如果能把工作量限制在局部，往往已经够快。

## 反模式警示

### 1. 先把所有东西挂出来，再靠 CSS 隐藏

这通常只是把问题从“看得见”变成“看不见”。

### 2. 一次性加载整棵树

IO、内存和渲染压力会同时放大。

### 3. 长列表不窗口化

只要 DOM 数量和数据量一起涨，后面很难补救。

### 4. 还没到万级数据，就先上重型虚拟化抽象

很多时候是在给未来假设交税。

### 5. 把细粒度框架写成粗粒度更新

框架支持局部更新，不代表你的状态组织天然就是局部的。

## 最小清单

- 不可见内容能不能不挂 DOM
- 数据能不能按展开或按窗口加载
- 长列表能不能先切窗口
- 一个状态变化会不会带着整片子树一起重跑
- 滚动时有没有高频布局写入
- 这个场景到底需不需要复杂通用库
