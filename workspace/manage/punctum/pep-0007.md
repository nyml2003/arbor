## Abstract

Punctum 不再从“统一 TUI 与 GUI 的框架”开始设计。系统先从一个真实、可运行、可验收的产品切片生长。公共抽象只能从两个实现的相同语义中提取。

本 PEP 采用以下顺序：

```text
产品事实
-> 用户场景与验收 oracle
-> 产品状态和 action
-> target 私有的完整实现
-> 第二个真实实现
-> 比较相同与不同
-> 抽取最小公共合同
-> 稳定 PEP 和 crate
```

GUI 与 TUI 是两条呈现路径。它们可以共享产品 core、规范化输入和经过验证的交互语义，但不预设共享 UI tree、布局单位、文本系统、paint primitive 或 renderer。

依赖关系仍必须形成 DAG。不过 DAG 是完成边界设计后的检查结果，不是预先拆 service 和 crate 的方法。DDD、port 和 mock 只用于存在领域状态或副作用的边界。

## Why This Reset Is Necessary

当前 Punctum 同时尝试解决：

- TUI 与 GUI 的共享边界。
- Flex 布局。
- Canvas、HUD 和 Z 顺序。
- Focus、按键路由和交互状态。
- Terminal Unicode。
- GPU 文字、图片和资源。
- Desktop、Web 和 Mobile 生命周期。
- crate/service DAG、DDD 和 mock。

这些问题单独都合理，但它们还没有被同一个真实页面约束。结果是每个设计都必须为其他未实现设计预留扩展点。复杂度来自未知问题之间的乘法，而不是来自现有产品。

现有代码提供了另一组更可靠的事实：

- `punctum-grid` 已能处理离散 grid、surface、diff 和 patch。
- `punctum-input` 已能规范化键盘事件。
- Terminal 已证明 grapheme、列宽和 continuation 是目标私有复杂度。
- GPU grid 已证明 tile quad 与普通 GUI 文字、图片不是同一种 primitive。
- `punctum-ui` 只证明了离散整数空间中的无状态布局形状。
- Game GUI、Focus、HUD 和跨 target 页面复用尚未被真实产品证明。

因此，继续补齐横向框架会扩大推测性设计。当前需要改变迭代单位，而不是继续增加专题 PEP。

## Goals

1. 用可运行的纵向场景代替横向框架建设作为迭代单位。
2. 分开事实、约束、问题、假设、证据和稳定合同。
3. 让 GUI 与 TUI 各自承担目标私有复杂度。
4. 只从相同语义和相同测试 oracle 中抽取公共能力。
5. 让 crate、service、DDD、port 和 mock 由真实边界触发。
6. 保留现有已验证能力，避免无证据的整体重写。

## Non-Goals

本文不定义 Flex、Focus、Text、Image、Canvas 或 Host 的最终公共 API，不决定完整组件目录，也不承诺 TUI 与 GUI 的代码复用比例。本文只定义它们获得公共地位前必须经过的迭代和证据。

本文也不改变 Battle 产品规则、Ramus capability、第一期玩家行为或现有平台支持范围。

## Interpretation Of The Mythical Man-Month

本 PEP 使用《人月神话》的四条工程判断。

### Conceptual Integrity

系统需要一个清楚、统一的概念模型。概念完整性不是让所有 target 使用相同类型，而是让每个概念只有一个含义和一个所有者。

例如：

- `GridPos` 只表示离散游戏空间。
- Terminal Cell 只表示终端列。
- GUI layout unit 只表示 GUI 虚拟像素。
- Product Action 只表示产品意图。

为了表面统一而创建万能 Cell，会破坏概念完整性。

### Second-System Effect

第二套系统容易一次补上第一套系统缺少的全部能力。Punctum 当前的风险正是把 Grid 基座扩张成同时覆盖 TUI、GUI、游戏、普通应用和所有平台的第二系统。

控制方法不是设计更多扩展点，而是限制一次迭代只解决一个用户场景，并删除不属于该场景的合同。

### Sharp Milestones

里程碑必须是可以判定真假的运行结果，不能是“框架基本完成”或“抽象已经通用”。

有效里程碑示例：

> 玩家可以在 Tetris GPU 中只用键盘打开 Ramus command palette，模糊筛选授权命令，执行后通过同一个 Tetris transition 改变游戏，并在 resize 后保持面板可用。

该里程碑可以运行、截图和测试。`FocusService` 或 `Flex crate` 完成不是产品里程碑。

### Grow, Do Not Rebuild

早期版本应允许被丢弃，但不应计划一次完整重写。实验代码放在产品边界内，成功后逐步提取，失败后直接删除。

“先写一个再扔掉”在这里表示允许丢弃 probe 的局部设计，不表示先造完整框架，再整体重写一次。

## Complexity Classification

### Essential Product Complexity

以下复杂度由产品目标产生，不能靠抽象消除：

- 对战状态、合法动作和结果。
- 键盘选择、确认、返回和 modal 行为。
- 当前选择、业务选中和 UI focus 的区别。
- 在窄窗口中保持关键选项可见。
- 游戏画面、HUD、日志和浮层的视觉顺序。

### Essential Target Complexity

以下复杂度由宿主能力产生，应由 target 自己承担：

- Terminal grapheme、CJK 列宽、continuation 和 ANSI patch。
- GUI shaping、glyph atlas、图片、整数虚拟像素和 GPU resource。
- surface acquire、resize、DPI、suspend、resume 和 present。
- Terminal 与 GUI 不同的文字测量和裁剪。

### Accidental Architecture Complexity

以下复杂度当前没有产品证据，应删除或延后：

- 万能 Cell 或跨 target paint command。
- 强制相同的 TUI/GUI tree。
- 每个能力先拆 crate/service。
- 每个 DAG 节点先发布 trait 以便 mock。
- 每个 service 强制使用完整 DDD 目录。
- 在真实 Button 或 Menu 出现前定义交互状态全集的存储结构。
- 在一个 GUI 页面出现前冻结通用 Flex、theme 和 image API。
- 把待研究问题全部升级为 PEP。

## Knowledge States

每项设计必须处于以下一种状态，不能混写：

| State | Meaning | Allowed action |
| --- | --- | --- |
| Fact | 已有代码、测试或产品决定证明 | 可以依赖 |
| Constraint | 当前产品或平台必须满足 | 必须验收 |
| Question | 尚未回答 | 记录，不创建公共 API |
| Hypothesis | 一个可能方案 | 在产品内 probe |
| Evidence | probe 的可观察结果 | 用于比较 |
| Decision | 已选择并说明代价 | 可以指导下一迭代 |
| Contract | 多个实现证明的稳定边界 | 可以进入公共 PEP |

“听起来合理”只能成为 Hypothesis。Draft PEP 也不能把 Hypothesis 当作实施依赖。

## New Logical Structure

### 1. Product Core

Product Core 拥有业务状态、不变量和 action。

```text
BattleState + BattleAction -> BattleOutcome
```

它不依赖 UI、Focus、Terminal、GPU、字体、图片或平台事件。

这是最稳定的共享部分。TUI、GUI、人类输入和 Agent 命令最终都调用同一个 application API。

### 2. Product Session

Product Session 保存完成当前用户流程所需的短期状态。例如：

- 当前菜单。
- 当前候选项。
- 已打开的 modal。
- 当前日志窗口位置。
- 尚未提交的 action draft。

这些状态如果影响产品行为，就由产品拥有。框架不能把它们藏进通用 Element tree。

UI Focus 是否属于 target session，取决于两个 target 是否具有相同导航图。首个 GUI probe 可以先在 `game-ui` 内实现。

### 3. Input Mapping

平台 adapter 只产生规范化输入事件。Host 按明确优先级映射事件：

```text
Platform Event
-> Canonical Key Event
-> global/modal/focused binding
-> Product Intent
-> Product Session or Application
```

物理键、逻辑键、repeat、press/release 和 consumed/unhandled 必须在真实流程中定义。Focus 不直接读取 winit 或 crossterm 事件。

### 4. Target View

TUI 与 GUI 各自从 Product Core 和 Product Session 构造只读 view。

```text
Product Snapshot
├─ Terminal View
└─ GUI View
```

两棵 tree 可以不同。GUI 可以有图片、Canvas 和动画。Terminal 可以有压缩信息和终端专用布局。产品不为了复用率维持节点一一对应。

共同 builder 只能在实际重复后抽取。抽取对象优先是产品语义和 action mapping，不是 target geometry。

### 5. Target Layout And Planning

每条路径只使用自己的单位和 primitive。

```text
Terminal View
-> terminal row/column layout
-> TerminalCell surface
-> diff/patch plan

GUI View
-> integer virtual-pixel layout
-> text/image/rect/canvas draw plan
-> GPU frame plan
```

GUI 第一版可以在 Game 内使用整数 Flex 子集和 Stack。所有 style geometry 使用整数。除不尽的空间按稳定顺序分配余数。

这条规则只先约束 GUI probe。除非 Terminal 出现相同算法和 oracle，否则不能因此发布跨 target Flex API。

### 6. Platform Driver And Host

Driver 只执行 target plan。Host 拥有：

- event loop。
- tick 和 redraw 顺序。
- surface/device 生命周期。
- 一帧一次 acquire、submit 和 present。
- suspend、resume、resize 和恢复策略。

Canvas、HUD 和 modal 只能向同一个 GUI frame plan 追加命令。它们不能各自 present。

### 7. Derived Dependency DAG

完成以上边界后，crate 依赖自然形成 DAG。评审 DAG 时只检查：

- Product Core 不依赖 target 和平台。
- Target View 不修改 Product Core。
- Layout/Planner 不拥有平台 surface。
- Driver 不包含产品状态转换。
- Host 是 composition root，不使用 Service Locator。

一个逻辑层不必对应一个 crate。出现独立发布、编译隔离、复用或测试价值后再拆 crate。

## Shared Boundary Policy

共享能力分为三档。

### Confirmed Shared

- 产品 domain/application 状态和 action。
- 已有的规范化键盘事件。
- 离散游戏 grid 需要的 geometry、surface 和 diff。
- 不包含 target 类型的只读产品 snapshot。

### Candidate Shared

- key binding 和 routing outcome。
- Focus intent、scope 和 reconciliation。
- 部分无状态容器语义。
- semantic style token。
- Text 的内容和 overflow policy。

候选能力先留在真实消费者内。两个 target 的输入、状态转换和 oracle 相同时才抽取。

### Target Private

- Terminal Cell、Unicode width、ANSI plan。
- GUI Flex metrics、字体 layout、Image、Canvas 和 draw command。
- Terminal GridView 与 GUI CanvasView payload。
- platform surface、DPI 和 present error。

Target Private 不表示代码质量较低，也不表示以后不能抽取。它表示当前没有足够证据承诺兼容。

## Iteration Unit

一次迭代必须是一个纵向用户场景，不是一个水平框架层。

### Step 1: Select One Scenario

场景必须同时包含输入、状态变化和可见输出。第一场景固定为：

> 在 Tetris GPU 中，玩家按 `Ctrl+P` 打开 Ramus command palette，输入查询、移动选择并执行授权命令；棋盘与面板在同一个 GPU frame 中绘制。

### Step 2: Define Oracles First

至少定义：

- 给定 principal，哪些 Ramus invocation 可见和可执行。
- 给定 query 和按键序列，候选与 selection 如何变化。
- 执行后产生哪个 `TetrisCommand`。
- 直接键盘和 Ramus 是否进入同一个 `transition`。
- 面板打开时 tick 是否暂停，关闭后是否避免追赶。
- 三个 viewport 下 query、候选和 diagnostic 是否可见。
- resize 前后产品和 palette semantic state 不变。
- 一帧只能 present 一次。

### Step 3: Build In The Product

所有新 palette 和 overlay 类型先写在 Tetris GPU example。只在 surface frame ownership 或 committed text 这类明确平台边界上修改 Punctum adapter。

第一版允许直接 enum、struct 和 match。允许重复少量 layout 或 routing 代码。禁止为了未来 target 建 registry、plugin、universal command 或 service wrapper。

### Step 4: Finish The Walking Skeleton

切片必须从真实键盘事件贯通到真实 GPU present。纯截图 mock 不能替代 host smoke。平台测试困难的部分使用 plan fixture，加一个真实运行入口。

### Step 5: Add Pressure Deliberately

同一实现依次承受：

- 三种 viewport。
- 空查询、长查询和零候选。
- 未授权命令。
- palette 打开、关闭和执行失败。
- tick 暂停与恢复。
- surface resize。

只添加会改变边界判断的压力，不做完整组件目录。

### Step 6: Implement The Second Consumer

Tetris command palette 完成后，再在 Game 中实现真实的 in-game Ramus console。第二个消费者可以使用不同 layout 和视觉，只比较 capability、interaction、execution 和 product action 的共同语义。

### Step 7: Extract Or Delete

比较两个实现：

- 相同语义、相同状态转换、相同 oracle：候选抽取。
- 名称相同但单位、错误或生命周期不同：保持 target private。
- 只为一个 target 服务：删除共享 wrapper。
- probe 没有进入产品路径：删除 probe。

### Step 8: Record A Sharp Milestone

迭代结束只记录：

- 玩家可运行的结果。
- 通过的 oracle。
- 新确认的事实。
- 被否定的假设。
- 下一项仍未知的问题。

不使用“基础设施完成度百分比”。

## Planned Iterations

### Iteration 0: Architecture Reset

- PEP 1 由本文替代。
- PEP 2 至 PEP 6 延后，保留为候选设计库存。
- `punctum-ui` 保持 provisional，不扩张 API。
- Tetris 保留为 grid、input 和 adapter 回归项目，不再承担证明通用 GUI 框架的职责。

完成标准：管理文档不再把实现 PEP 2 至 PEP 6 当作进入 Game GUI 的前置条件。

### Iteration 1: Tetris GPU Ramus Command Palette

- 建立 capability-filtered Ramus command candidates。
- 建立纯 PaletteState 和模糊筛选。
- 补齐 committed text 和单帧 grid/overlay composition。
- 键盘事件贯通到 Ramus runtime 和现有 `TetrisCommand`。
- 完成一个真实 GPU host frame。

完成标准：命令面板可运行；授权、暂停、resize、同一 transition 和一次 present oracle 通过。

### Iteration 2: Host And Text Pressure

- 明确 repeat、release、consumed/unhandled。
- 明确 redraw、tick、resize、skip 和 present commit。
- 加入 CJK/ASCII、IME commit、光标编辑边界和长 diagnostic。
- 验证 surface 丢失或暂停后的恢复边界。

完成标准：measure 与 paint 使用同一 text layout；跳过的 frame 不污染已提交状态。

### Iteration 3: Game In-Game Ramus Console

- 在真实 Game host 中实现 capability-filtered console/palette。
- 使用 Battle application action，不复制 Tetris command vocabulary。
- 验证图片、HUD、日志和 console overlay 共用一个 GPU frame。
- 比较两个产品的 palette state、input routing 和 execution oracle。

完成标准：Tetris 和 Game 的相同语义可以由同一 black-box suite 验证；视觉和产品 action 保持产品私有。

### Iteration 4: Extraction Review

- 对比 Tetris 与 Game 的 input、palette、Ramus projection、layout 和 text 合同。
- 抽取相同语义，删除只转发的 wrapper。
- 决定现有 `punctum-ui` 是保留为 Terminal/Grid UI、重构还是停止扩张。
- 需要长期兼容的部分再写专题 PEP。

完成标准：每个公共类型都有两个真实调用点或一个不可跨越的平台边界证据。

## Rules For Keeping Each Layer Simple

1. 每层只拥有一种状态来源。
2. 每层只使用一种坐标单位。
3. 每层只输出下一层需要的词汇。
4. 只允许相邻层转换，不跨层传 backend handle。
5. 纯计算默认直接调用，不为 mock 创建 trait。
6. 只有 IO、时钟、资源和 driver 使用 port。
7. 产品 action 不包含 FocusId、LayoutRect 或 KeyCode。
8. View 不修改产品状态，只产生 intent/action。
9. Planner 不 acquire、submit 或 present。
10. Driver 不理解 Battle、Button 或 Canvas 语义。
11. 错误由能处理它的最近边界转换，不建立万能错误 enum。
12. 同一概念出现第二个真实实现前，不拆公共 crate。

## Validation

### Product Tests

- Tetris direct key 和 Ramus invocation 比较最终 `TetrisCommand` 与状态。
- Palette/Product Session 使用语义 intent 序列测试。
- 不同产品或 target 比较最终 Product Action，不比较 rect 或截图相同。

### Target Tests

- GUI 使用 layout/draw plan fixture 和关键截图。
- Terminal 使用 cell surface、continuation 和 patch fixture。
- Text 分别使用 target 自己的 measure/paint 一致性 oracle。

### Host Tests

- 平台事件规范化使用 fixture。
- frame commit、resize 和恢复使用状态机测试。
- 每条真实平台路径保留最小 smoke。

### Extraction Gate

公共抽取必须回答：

1. 哪两处真实代码将被删除。
2. 两处是否有相同 input alphabet。
3. 两处是否有相同 state transition。
4. 同一 black-box oracle 是否可以原样运行。
5. 抽取后是否新增 target 分支或 optional 字段。

第 2 至第 4 项任一为否，默认不抽取。第 5 项为是时，应优先缩小合同。

## Backwards Compatibility

Punctum 当前没有已发布的稳定 GUI、Focus 或 widget API。本文不要求破坏现有 grid、input、Terminal、GPU grid 或 Tetris 行为。

现有 `punctum-ui` 保持 provisional。已有消费者可以继续使用，但不得把它的 `GridSize` 布局解释为 Native GUI 稳定合同。后续决定删除或重构该 crate 时，必须先保留现有 fixture 和消费者行为，再单独执行迁移。

PEP 2 至 PEP 6 均未进入 Accepted，因此延后不产生稳定 API 兼容义务。

## Migration

1. 不删除现有 Punctum crate 和已通过测试。
2. 停止把 `punctum-ui` 当作 Native GUI 的既定基础。
3. 第一条 GUI probe 归 Tetris GPU example 所有；第二个消费者归 `game-ui` 和 `game-host` 所有。
4. 现有 Tetris Terminal/GPU 保持为 grid/adapter 回归入口。
5. Game GUI 完成后再决定是否复用、替换或收窄 `punctum-ui`。
6. PEP 2 至 PEP 6 的规则只能按真实证据逐项恢复，不能整包实现。
7. 依赖图继续保持无环，但不按假想 service 预建 crate。

## Rejected Alternatives

### Finish All PEPs Before Coding

这会让每个 PEP 为其他未实现 PEP 预留能力，继续扩大第二系统效应。

### Rewrite Punctum From Scratch

现有 grid、input、Terminal Unicode 和 GPU grid 已有验证价值。整体重写会丢失证据，并把架构问题变成迁移问题。

### One Universal UI Tree

它会把 GUI Image、Canvas、字体和 Terminal continuation 合并为字段并集。共享产品 action 比共享节点形状更稳定。

### Service Per Capability First

Service、crate、port 和 mock 都有维护成本。它们应由状态、IO、复用和测试隔离需求触发。

### GUI Only, Terminal Later Without A Gate

这会让 GUI 假设悄悄成为公共合同。Terminal 可以后做，但必须在公共抽取前作为第二条证据路径。

### Preserve Every Draft For Compatibility

Draft 没有兼容性承诺。保留错误前提会让后续实现同时满足新旧模型。旧 Draft 应保留历史，但退出依赖链。

## Relationship To Earlier PEPs

- PEP 1 被本文替代。其问题分析仍可作为背景。
- PEP 2 的整数 GUI layout 和余数规则是 Iteration 1 的候选实现，不是公共合同。
- PEP 3 拆成 GUI draw order、Terminal overlay 和 Host frame commit 三个问题。
- PEP 4 的 Focus 状态机先作为 Game-owned interaction probe。
- PEP 5 的 target-specific leaf 判断保留，但不先发布 custom-leaf API。
- PEP 6 作为语义词汇库存，按组件需求逐轴引入。

## Open Issues

1. WGPU frame composition API 在 Tetris probe 后是否足够稳定，仍需 Game 第二个消费者验证。
2. Palette query 是否在 Iteration 2 扩展为完整 Ramus shell text，必须由参数编辑需求决定。
3. Tetris 和 Game 的 palette state 是否同构，必须通过共同 black-box suite 判断。
4. Terminal Ramus console 是否需要进入后续抽取 gate，当前不作为第一期前置条件。
5. 现有 `punctum-ui` 最终定位只能在 Iteration 4 决定。
