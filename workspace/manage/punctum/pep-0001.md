## Status Note

PEP 7 替代本文作为总体架构。本文保留原始问题、候选边界和否决方案，但不再提供实施顺序或公共合同。特别是，共享 UI tree、本地 Service DAG 和“能力先成为 service”不再作为架构起点。

## Abstract

Punctum 采用一套 keyboard-first UI 架构。公共 UI 使用 FlexBox 子集、Stack、Focus 和 Action。GUI 使用 `CanvasView` 嵌入游戏世界。Terminal 使用自己的 `GridView` 投影游戏区域。

TUI 与 GUI 共享容器协议、布局算法、Focus 和适用于双方的语义样式。它们不要求使用相同叶子集合或构造完全相同的 tree。图片是 GUI 能力，不进入 TUI 合同。

Punctum 在本地进程内按能力服务组织。每个稳定能力可以成为一个 crate/service。服务依赖构成有向无环图（DAG）。服务通过纯值合同或 port 可替换测试，但不要求每个 crate 包一层 trait，也不引入网络微服务或 Service Locator。

## Motivation

当前实现已经证明以下能力有效：

- `punctum-grid` 可以表达离散坐标、`Surface<T>`、diff 和 patch。
- `punctum-input` 可以统一键盘事件和 committed text。
- `punctum-terminal` 可以处理 grapheme、Unicode width 和 continuation。
- `punctum-gpu` 可以把固定 grid 投影为等尺寸 sprite instance。
- `punctum-ui` 已验证 measure、layout、paint 和 clip 的纯算法形状。

当前实现也暴露了原假设的边界：

- 一个 Tetris 逻辑格在 Terminal 中投影为两列，在 GPU 中投影为一个方形 quad。
- `GpuCell` 只能表达固定 grid sprite，不能表达普通文字、图片和 HUD 矩形。
- `punctum-ui` 使用 `GridSize/GridRect` 作为全部布局几何，混淆 grid slot、Terminal 行列和 GUI 虚拟像素。
- Tetris 和 Game 尚未真实消费 `punctum-ui`。
- 当前 GPU runtime 在一次 `present` 中独占 surface 获取、清屏、grid draw 和提交，无法继续叠加 HUD。

因此，Punctum 不能继续把 Cell 当作跨目标统一单位。需要统一的是 UI 编译流程和键盘交互语义。

## Goals

本 PEP 建立以下合同：

1. 一套容器组合协议、FlexBox 子集、Stack、Focus 和 Action 模型。
2. custom leaf 作为页面 layout 与 target-specific 游戏绘制的明确边界。
3. TUI 与 GUI 各自拥有叶子测量、绘制计划和平台提交。
4. keyboard-first Focus 不依赖 hit testing 或布局坐标。
5. 能力服务构成本地 crate DAG，并可以按节点性质替换测试。
6. 服务内部按实际领域复杂度采用 DDD，不强制空洞分层。
7. 首版允许每帧重建、布局和绘制，不复制 Flutter 的 Element/reconciliation 复杂度。

## Non-Goals

第一阶段不包含：

- 鼠标、触摸、hover、pointer capture 或 gesture arena。
- 文本编辑器、IME composition、选区和光标。
- 完整 CSS FlexBox、wrap、percentage 或 intrinsic 多次测量。
- Flutter 式 Widget/Element/RenderObject 三树和增量失效系统。
- 通用滚动、虚拟列表、任意 blur、filter 或离屏 layer。
- 网络微服务、RPC、服务发现或运行时 Service Locator。
- 为每个 crate 强制创建 trait、Repository、Aggregate 或 application 文件夹。
- 让产品状态先转换成通用 `ScreenModel`。
- 为了 tree 形状一致，把 GUI-only 能力塞进 TUI 类型系统。

## Terminology

### Capability Service

能力服务是一组具有独立词汇、不变量、输入输出和测试合同的本地能力。服务通常映射为一个 Rust crate，但小型内部实现可以先保留为模块。

服务不是线程、进程或网络微服务。服务之间通过 Rust 类型、函数和显式 port 调用。

### Port

Port 是核心为了调用可变行为或外部能力而定义的接口。平台 adapter、字体系统、资源加载和 GPU driver 可以实现 port。

纯确定性 crate 之间可以直接调用，不需要为 mock 人为增加 port。

### Test Substitute

替身包括真实纯实现、内存实现、fake、stub 和 mock。测试应选择成本最低且仍能验证合同的替身。

“节点可替换”表示 composition root 可以换掉该能力的实现，不表示每个函数都必须动态派发。

### Horizontal Layer

水平层描述一次 UI 帧从产品状态到平台提交的阶段。依赖只能向下。

### Vertical Capability

垂直能力描述一个概念穿过多个水平层的完整路径，例如 Focus、Text、游戏叶子或 Frame。

### Target

Target 是一次布局和绘制的目标环境。第一阶段包含 Terminal target 和 GPU target。

### CanvasView

`CanvasView` 是 GUI tree 中的 target-specific 游戏叶子。Terminal 对应角色是 `GridView`。两者只共享 custom-leaf 外部 layout/clip/focus 合同，不共享 payload 或 paint command。

## Core Principles

### Abstraction Admission

公共抽象必须同时满足：

1. 有独立术语和可测试不变量。
2. 至少两个真实使用点需要相同合同，或一个使用点已经证明平台边界必须隔离。
3. 抽象不会要求 Terminal、GPU、Web 或产品模型伪装成相同实现。
4. 删除抽象后会产生真实重复或依赖反转，而不只是目录不对称。

发现抽象泄漏时，优先缩小合同、退回 target/产品模块或删除抽象。兼容既有草案不优先于保持边界合理。

### Keyboard First

Focus 是 Punctum 的核心交互模型。平台键盘事件先转换为有限 `FocusIntent`，再进入 Focus 和产品 Action。

```text
KeyEvent
-> KeyBinding
-> FocusIntent
-> FocusService
-> FocusOutcome
-> Product Action
```

### GUI Outside, Canvas Inside

整个页面由 GUI tree 组织。游戏内容不是另一套页面框架，而是 GUI 的 `CanvasView` 叶子。

```text
Ui
└─ Stack
   ├─ CanvasView(game scene)
   ├─ SafeArea
   │  └─ Flex(HUD)
   └─ ModalOverlay
```

### Shared Protocol, Target-Appropriate Leaves

TUI 与 GUI 共享有真实双端消费者的结构和算法，不共享最终几何结果，也不强求相同叶子集合。

共享内容：

- Flex、Stack、Padding 和 Align。
- Focus scope、顺序、方向和 Activate/Back。
- Action 和 enabled/visible 语义。
- 双方实际使用的 Text 语义。
- 游戏叶子的外部 bounds、clip 和 focus 语义。
- 双方实际使用的语义 style token。

目标私有内容：

- Terminal grapheme width、continuation 和 ANSI plan。
- GPU glyph shaping、GUI-only Image、virtual pixel 和 draw list。
- 目标 spacing、border 和字体 metrics。
- 平台 surface、DPI、safe area 和 present。

产品可以复用 common component builder，再分别构造 Terminal view 和 GUI view。共享比例由真实产品决定，不设“tree 必须相同”的指标。

### One Coordinate Space Per Layer

每个类型只能表达一个坐标空间：

- `GridPos/GridSize/GridRect`：离散 world/grid。
- `LayoutPos/LayoutSize/LayoutRect`：当前 target 的布局空间。
- `PixelSize/PixelRect`：framebuffer 物理像素。
- GridView/CanvasView 内部坐标由各 target leaf 拥有。

禁止使用 `GridSize` 表达 GUI layout。禁止跨 target 比较 `LayoutRect`。

## Local Service DAG

箭头 `A -> B` 表示 A 依赖 B。

```text
product-view ------> product-core
     |
     +-------------> punctum-ui -------> punctum-focus

product-host ------> product-core
     |
     +-------------> product-view
     +-------------> punctum-input
     +-------------> punctum-terminal -> punctum-grid
     |                    |
     |                    +------------> punctum-ui
     +-------------> punctum-gpu ------> punctum-grid
                          |
                          +------------> punctum-ui

terminal-host -----> punctum-crossterm -> punctum-terminal
                              |
                              +--------> punctum-input

gpu-host ----------> punctum-wgpu ------> punctum-gpu
                              |
                              +--------> punctum-input
```

`product-core` 不依赖 Punctum。`product-view` 读取产品公开状态并构造 UI tree。`product-host` 负责把 Focus outcome 或快捷键映射为产品 action。

`punctum-focus` 不依赖布局和 backend。`punctum-ui` 可以引用 Focus ID，并提供 custom-leaf 外部布局入口，但不定义共享 Canvas payload，也不依赖 Terminal、wgpu、winit 或产品类型。Focus API 没有达到独立 crate 门槛前可以留在 `punctum-ui` 模块中。

`punctum-terminal` 和 `punctum-gpu` 是 target planning service。平台 crate 只执行 plan 和规范化平台输入。

### DAG Rules

1. crate graph 必须无环。
2. 产品 core 不依赖 UI 或平台 crate。
3. target service 可以依赖共享 UI，不允许共享 UI 反向依赖 target。
4. pure planning 与平台 IO 必须位于不同 crate。
5. host 是唯一 composition root。service 不通过全局注册表查找其他 service。
6. 跨 service 的 ID 使用 newtype，不能混用裸整数或字符串。
7. 一个新 crate 必须拥有独立合同和验证价值，不能只为目录对称而创建。

### Compile-Time And Runtime Graphs

crate DAG 是编译期依赖图。它约束代码可见性、类型所有权和依赖方向。纯算法 crate 通常只存在于这张图中。

service instance graph 是运行期对象图。它只包含具有状态、资源或外部副作用的实例，例如 Focus state、字体资源、GPU context 和 Terminal driver。Host 在 composition root 显式构造并注入这些实例。

不能因为一个 crate 是 DAG 节点，就强制它拥有运行期对象、trait object 或生命周期。也不能因为两个能力位于同一 crate，就允许它们绕过各自合同。

测试替换发生在最窄消费边界：

- 纯 crate 默认直接使用真实实现和固定输入。
- consumer 确实需要隔离某个纯能力时，由 consumer 定义窄 port，并在测试中注入 fake。
- 有状态或副作用 service 通过构造参数注入实现。
- Host 可以用 test composition root 替换任意需要隔离的节点。

因此，“任意节点可 mock”是隔离能力，不是统一实现机制。架构要求任何节点都不能通过全局单例、隐藏 IO 或反向回调变成不可替换；不要求 provider 为所有 API 预先发布一对一 wrapper trait。

## Service And DDD Policy

每个能力服务内部按领域复杂度选择 DDD 强度。

| Service | 领域复杂度 | 推荐结构 |
| --- | --- | --- |
| Grid | 低 | value object + pure functions |
| Input | 低 | canonical event model + platform adapters |
| Focus | 中 | aggregate/state machine + structured outcome |
| UI/Layout | 中 | node model + constraints + pure layout service |
| Terminal target | 中 | terminal model + planner + driver port |
| GPU target | 中 | render plan + resource identity + driver port |
| Platform host | 低 | adapter + composition root，无领域模型 |

DDD 规则：

1. 服务先定义自己的统一语言和不变量。
2. 有生命周期和状态转换时使用 aggregate/state machine。
3. 外部能力通过由核心需要定义的 port 进入。
4. 正常失败使用结构化 `Result`。
5. 没有领域状态时使用纯函数，不创建假的 Aggregate、Repository 或 Domain Service。
6. application 层只在一个服务需要编排多个领域操作时出现。
7. adapter 不把平台错误码泄漏到共享核心。

“每个能力是服务”是所有权原则，不是要求每个能力都使用完整战术 DDD 模板。

## Substitutability And Mock Policy

每个 DAG 节点必须可在 composition root 被替换，但替换方式取决于节点性质。

| Node kind | 默认测试方式 | 是否需要 mock port |
| --- | --- | --- |
| 纯 value/algorithm crate | 使用真实实现 | 否 |
| 有状态纯服务 | 使用真实内存状态 | 通常否 |
| 资源、字体或时钟 port | fake/in-memory implementation | 是 |
| Terminal/GPU driver | plan oracle + mock/fake driver | 是 |
| 平台 host | fixture、smoke、E2E | 不强制单元 mock |

禁止以下模式：

- 为 `Surface::diff` 创建 `GridService` trait 只为了 mock。
- provider 同时发布 concrete API 和一层无语义 wrapper trait。
- 在运行时使用字符串查找 service。
- mock 平台对象后声称验证了真实 GPU、Terminal 或生命周期。

推荐由消费方为自己的需要定义窄 port。例如，UI target 只需要 `TextMeasurer`，不应依赖完整字体引擎 API。

## Horizontal Layers

### Layer 0: Product State

产品 core 持有游戏或工具状态。Punctum 不规定 Domain、Application 或 ScreenModel。

### Layer 1: UI Description And Interaction

产品 view 构造 `Node` tree，绑定 `FocusId`、enabled、visible 和产品 action mapping。

该层不包含 target 坐标、glyph、Terminal Cell、GPU resource 或平台事件。

### Layer 2: Measure And Layout

Target 提供叶子测量和 style metrics。共享 layout service 运行 Flex、Stack、Padding、Align 和 clip，产生 placed frame。

一次 layout pass 只使用一种 target-local `LayoutUnit`。

### Layer 3: Target Planning

Terminal target 把 placed frame 编译为 Terminal frame/patch plan。GPU target 把 placed frame 编译为有序 draw plan。

### Layer 4: Platform Execution

Crossterm、wgpu、Web、Android 或 iOS adapter 执行 plan，处理 surface 生命周期、DPI、safe area 和平台错误。

### Layer 5: Host Composition

Host 拥有 event loop、service 实例、frame 生命周期和产品 action 调用。其他 service 不能创建 host 或反向调用产品状态。

## Vertical Capabilities

### Focus Capability

Focus 是不依赖 layout/backend 的纯状态机。方向关系来自逻辑结构或显式 neighbor，不根据像素或 Cell bounds 推断。完整合同见 PEP 4。

Focused、Selected、Active 和 Pressed 是正交状态。Focus service 只拥有 Focused。完整状态集合见 PEP 6。

### Layout Capability

Punctum 使用单行 Flex 子集和整数 target-local layout。grow、shrink、无法整除的余数、overflow 和测量次数由 PEP 2 定义。覆盖不进入 Flex 算法。

### Text Capability

共享 `Text` 节点只保存内容、语义 style 和 wrap policy。

Terminal target 负责 grapheme、Unicode width、continuation、cell clip 和 fallback。

GPU target 使用成熟 shaping/rasterization 实现，负责 glyph、font fallback、baseline 和 glyph atlas。不得假设 char、grapheme 和 glyph 一一对应。

Measure 与 paint 必须复用同一份 text layout 结果。

### GUI Image Capability

`Image` 是 GUI target 的叶子，不属于 TUI/shared UI 合同。它保存 resource identity、fit 和可选 accessibility description。source rect 与 destination rect 分离。

GPU target 输出 image quad。Terminal view 不构造 Image，也不需要 fallback、图片协议或 unsupported capability。产品确实需要 Terminal 文字说明时，应显式构造 Text，而不是让 Image 在 TUI 中变形。

### Game Leaf Capability

Terminal `GridView` 与 GUI `CanvasView` 是不同 target leaf。它们只共享外部 bounds、clip 和 focus 行为，不共享 payload、resource 或 paint command。完整合同见 PEP 5。

### Frame Capability

Terminal frame：

```text
placed UI
-> Surface<TerminalCell>
-> Patch<TerminalCell>
-> TerminalPlan
-> Crossterm
```

GPU frame：

```text
acquire frame once
-> clear
-> Canvas commands
-> HUD commands
-> overlay commands
-> submit once
-> present once
```

GPU runtime 应拆为 frame owner 与独立 renderer：

```text
GpuContext / GpuFrame
├─ CanvasRenderer
├─ GuiRenderer
└─ present
```

## TUI And GUI Target Contract

| Concern | Shared contract | TUI private | GUI private |
| --- | --- | --- | --- |
| Tree | Flex/Stack/container protocol | Terminal leaf set | GUI leaf set |
| Focus | scope/graph/outcome | focus glyph/style | focus border/style |
| Layout | constraints/algorithm | row/column metrics | virtual-pixel metrics |
| Text | content/style | width/continuation | shaping/glyph atlas |
| Image | 无 | 不建模 | Image node/resource/quad |
| Game leaf | bounds/clip/focus | GridView/cell projection | CanvasView/viewport/draw commands |
| Output | ordered paint | Terminal frame/patch | GPU draw plan |
| Runtime | lifecycle result | ANSI/session | surface/device/present |

只有产品明确声明为双 target 的 view flow 才运行跨 target 合同。合同只断言：

- 共享流程显示相同语义内容；target-only 装饰不参与比较。
- enabled、visible 和 focus 顺序一致。
- Activate/Back 产生相同产品 action。
- 所有节点有界且 clip 合法。
- 游戏叶子获得合法 bounds。

跨 target 合同不断言：

- 相同文字宽度或换行。
- 相同布局矩形。
- 相同 padding 数值。
- 相同截图、patch 数或 draw call 数。

## Cross-Platform Contract

平台 host 提供：

```text
viewport
scale factor
safe-area insets
surface lifecycle
canonical input events
```

平台差异不能进入产品状态或共享 UI node。

### Terminal

Crossterm adapter 负责 raw mode、alternate screen、ANSI、resize 和平台输入。`TerminalCell` 严格表示一个终端列。游戏 tile 到多个终端列的映射属于 GridView，不属于 presenter 的全局 `cell_width`。

### Desktop GPU

winit host 负责 window lifecycle 和 input。wgpu adapter 负责 surface、device、queue 和 frame execution。UI 使用整数虚拟像素，host 映射到物理像素。

### Web

Web 页面把 GPU Canvas 当作宿主矩形。DOM 可以承载系统输入或外部页面，Punctum Canvas/HUD 在同一 GPU surface 中绘制。CSS size、bitmap size 和 device pixel ratio 由 Web host 转换。

### Android And iOS

Native 页面拥有 `SurfaceView/TextureView/MTKView/CAMetalLayer` 等宿主。Punctum renderer 接收 surface 和 lifecycle，不创建页面。

第一阶段只承诺外接键盘或 gamepad 语义。触摸需要新的 PEP。支付、登录、IME 和系统可访问性控件可以使用 Native overlay。

## Error Model

共享错误必须结构化：

- invalid constraints。
- duplicate Focus ID。
- missing focus target。
- unsupported target capability。
- missing resource。
- layout overflow。
- frame/surface lifecycle failure。

Platform adapter 在边界把原始错误转换为 target/runtime error。共享 UI 不返回 wgpu、winit、Crossterm、Android 或 iOS 错误类型。

## Performance Model

第一阶段允许每帧：

```text
rebuild UI tree
-> measure/layout
-> build target plan
-> render
```

优先缓存：

- text shaping result。
- glyph atlas。
- image texture。
- immutable target game resource。

在 profiler 证明必要前，不增加 Element tree、dirty layout、dirty paint 或局部 reconciliation。

GPU target 避免无界 `saveLayer`、blur、复杂 mask 和嵌套 opacity。clip 首选矩形 scissor。

## Testing Strategy

### Service Contract Tests

每个 pure service 使用真实实现测试输入、输出、不变量和结构化失败。

### Port Tests

字体、资源、driver 和平台 port 使用 in-memory fake 或窄 mock。provider 和 adapter 必须共享同一组 contract fixture。

### DAG Tests

- 检查 crate dependency 无环。
- 检查 shared UI 不依赖 target/platform crate。
- 检查产品 core 不依赖 Punctum UI 或平台 crate。
- 检查 target planner 不执行真实平台 IO。

### Cross-Target Tests

同一产品流程在 Terminal 与 GPU view 上验证共享语义内容、focus graph 和 action。各 target 使用自己的 tree fixture、golden 和 bounds oracle。测试不要求两个 tree 节点一一对应。

### Platform Tests

Terminal session、window lifecycle、surface、device 和 GPU submission 使用合同测试、smoke 和 E2E，不用纯 mock 替代真实平台验证。

## Current Crate Disposition

### Keep

- `punctum-grid`：继续拥有离散 geometry、Surface 和 patch。
- `punctum-input`：继续拥有 canonical keyboard/text event。
- `punctum-terminal`：保留 grapheme、continuation 和 terminal planning。
- `punctum-crossterm`：保留 Terminal platform adapter。

### Refine

- `punctum-ui`：保留 measure/layout/paint 形状；把 UI geometry 从 `Grid*` 改为 `Layout*`；增加 Flex、Stack、custom-leaf 和 Focus integration。
- `punctum-gpu`：把现有 `GpuCell`、grid viewport 和 patch planning 明确收窄为 Canvas/Grid renderer；增加 GUI draw plan 时不能扩张成万能 Cell。
- `punctum-wgpu`：拆分 frame ownership、grid/canvas renderer 和 GUI renderer；一次 frame 只 present 一次。
- Terminal presenter：移除或重新定义全局 `cell_width`；一个 `TerminalCell` 必须对应一个终端列。

### Add When Accepted

- `punctum-focus`：纯 Focus aggregate/state machine。如果 API 仍很小，可以先作为 `punctum-ui` 模块，达到独立依赖和复用门槛后再拆 crate。
- GPU text/image resource service：先在真实 Game/Tetris consumer 中验证，再决定是否拆 crate。

## Backwards Compatibility

Punctum crate 当前版本为 `0.0.0`，尚未建立外部稳定兼容承诺。本 PEP 允许内部 breaking change，但迁移必须保持：

- Tetris 业务状态和规则不变。
- Terminal Unicode 行为不回退。
- grid diff/apply 合同不变。
- canonical input 行为不变。
- GPU grid logical oracle 不变。

## Security And Reliability

- 图片、字体和 atlas 尺寸必须检查容量溢出。
- Focus ID 重复必须返回错误，不能静默覆盖。
- GridView/CanvasView clip 必须限制在分配 bounds 内。
- Platform surface 丢失、最小化和 resize 必须是正常结果，不使用 panic。
- 产品 action 只能由当前 enabled focus target 激活。

## Rejected Alternatives

### Universal Cell

拒绝把 Text、Image、Sprite、Continuation 和 HUD 全部压入同一个 Cell。它混合了不同坐标和生命周期。

### Separate Complete TUI And GUI Frameworks

拒绝复制组件树、Flex、Focus 和 Action。两端应共享语义管线，只分离目标测量和绘制。

### Full Flutter Clone

拒绝初期实现 Element tree、reconciliation、intrinsic layout、gesture arena 和增量 invalidation。游戏帧循环允许更简单的全量 rebuild。

### Every Crate Behind A Mock Trait

拒绝为所有纯 crate 添加接口包装。可替换性通过纯合同、内存实现和窄 port 实现。

### Service Locator Or Microservices

拒绝运行时字符串查找、全局容器、RPC 和分布式部署。Service DAG 是本地编译期架构。

### Mandatory Full DDD In Every Service

拒绝没有领域状态时仍创建 Aggregate、Repository 和 application shell。DDD 按复杂度使用。

### Generic Product ScreenModel

拒绝要求产品状态先转换成 Punctum 定义的通用 ScreenModel。产品 view 可以直接读取自身公开状态并构造 UI tree。
