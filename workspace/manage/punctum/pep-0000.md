## Abstract

本文定义 Punctum Enhancement Proposal（PEP）的编号、状态、评审和索引规则。

Punctum PEP 使用 Python PEP 的元数据和章节结构。PEP 用于记录长期架构、公共合同、兼容性和实施门禁。一次性任务进度仍写入 `workspace/manage/punctum-ramus-execution-plan.md`。

## PEP Types

- `Standards Track`：公共 API、架构边界、协议或可执行规范。
- `Informational`：设计说明、经验或背景，不建立强制合同。
- `Process`：PEP 流程、治理和发布规则。

## PEP Status

- `Draft`：正在讨论，不能作为稳定实现合同。
- `Accepted`：已批准，可以进入实施。
- `Active`：流程或长期规则正在生效。
- `Final`：实现和验证已经完成。
- `Deferred`：暂缓，保留编号。
- `Rejected`：已否决，保留原因。
- `Withdrawn`：作者撤回。
- `Superseded`：已由其他 PEP 替代。

## Lifecycle

1. 新 PEP 使用下一个可用编号。
2. `Draft` 必须包含动机、规范、兼容性、测试、否决方案和开放问题。
3. 架构 PEP 在真实消费者或明确 fixture 证明边界前不能进入 `Final`。
4. `Accepted` 只表示方案获准实施，不表示实现完成。
5. 实现偏离 PEP 时，先修改 PEP 或新建替代 PEP，不能只修改代码。
6. PEP 状态变化必须更新本文索引。

## Evidence Before Standards PEP

PEP 不再承担早期问题枚举和方案发散。一个公共合同按以下顺序生长：

1. `Question`：记录还没有答案的问题。它没有 PEP 编号。
2. `Probe`：在真实产品内实现可删除的实验。类型默认归产品或 target 所有。
3. `Evidence`：记录输入、输出、失败、重复和测试结果。
4. `Decision`：决定保留、删除或抽取。长期选择可以进入 PEP。
5. `Standard`：只有公共合同和兼容性承诺稳定后，才进入 Standards Track PEP。

单个实现可以证明平台边界必须隔离，但不能单独证明两个 target 应共享一个公共抽象。候选 API 在两个实现中出现，不等于它已经稳定；还需要证明两端具有相同语义、状态转换和测试 oracle。

Probe 应优先写在消费产品内。只有抽取能够删除真实重复，并且不会增加 target 无关的分支，才允许迁入 Punctum 公共 crate。

## Review Rule

PEP 的评审重点是可观察合同和依赖方向，不以目录数量、抽象数量或代码去重率作为通过标准。

专题 PEP 必须通过抽象泄漏审查：

- 共享概念是否至少有两个真实消费者。
- 是否只共享不变量，而不是把两端字段做并集。
- 是否迫使某个 target 实现没有需求的能力。
- 能否先保留为产品或 target 模块。

未通过时应缩小、延后或删除提案，不能为了编号连续而接受。

涉及以下事项时必须回到用户确认：

- 产品范围或玩家可见行为。
- 输入方式新增鼠标、触摸或文本编辑。
- TUI 与 GUI 的兼容承诺扩大或缩小。
- 公共 crate 的稳定性承诺。

不改变产品行为的 crate 拆分、类型命名、测试策略和内部算法可以由架构评审决定。
