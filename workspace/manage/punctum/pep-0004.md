## Status Note

PEP 7 要求先在真实 keyboard-first GUI 切片中验证 key binding、focus navigation 和产品 action 的边界。本文保留状态机候选，不要求先建立公共 focus service 或独立 crate。

## Abstract

本文定义 Punctum 的 keyboard-first Focus 合同。

平台按键先映射为有限 `FocusIntent`。Focus 状态机根据当前 scope、focus target 和显式导航关系，产生 focus change、intent dispatch、scope exit 或 unhandled outcome。

Focus 不依赖 layout bounds、Terminal Cell、GPU 像素、Z 顺序、鼠标或触摸。Canvas 可以显式 capture 方向 intent，把方向键交给游戏；未 capture 的方向 intent 才用于 focus navigation。

## Motivation

即使完全不支持点击和触屏，键盘 UI 仍然需要处理：

- 当前 focus 是谁。
- Tab/Shift-Tab 顺序。
- 方向导航。
- disabled/hidden 节点。
- modal focus trap。
- modal 关闭后的 focus 恢复。
- Canvas 使用方向键时如何阻止 focus 移动。
- UI 每帧重建后如何保留 focus。

如果这些规则分散在组件和 host 中，TUI 与 GUI 会产生不同交互。Focus 应是一个纯状态机，但不应扩张成通用 DOM event system。

## Goals

1. 为 TUI 和 GUI 提供同一套键盘 focus 行为。
2. 让 focus 与 target geometry 解耦。
3. 支持稳定遍历、方向导航、modal trap 和恢复。
4. 支持 Button、Menu、List 和 Canvas 的不同 intent capture。
5. 让每帧 rebuild 后的 focus reconciliation 可预测。
6. 使用纯输入输出测试全部状态转换。

## Non-Goals

第一阶段不支持：

- pointer focus、hover 和 hit testing。
- touch focus 和 gesture。
- DOM capture/bubble event phases。
- IME、文本选区和编辑光标。
- 根据像素距离自动猜测方向邻居。
- accessibility tree 的平台桥接。
- 多用户或多键盘的多个同时 focus。

## Minimal Public Model

规范只要求以下公共概念：

```text
FocusId
FocusScopeId
FocusIntent
FocusPolicy
FocusOutcome
```

Registry、graph、scope stack 和 state 可以由一个内部结构实现。本文不要求每个概念变成独立 trait、service 或 crate。

只有 Focus API 被 TUI、GUI 和至少一个产品稳定消费后，才考虑拆出 `punctum-focus` crate。此前可以作为 `punctum-ui` 的纯模块。

## Identity

`FocusId` 和 `FocusScopeId` 是不同 newtype，不能混用裸整数或字符串。

ID 必须在语义上稳定。每帧 rebuild 时，同一个产品控件应产生相同 ID。禁止使用当前 tree index 或内存地址作为跨帧 ID。

ID 只要求在当前 UI tree 内唯一。重复 ID 是结构化错误。

## Focus Intent

第一阶段 intent：

```text
FocusNext
FocusPrevious
MoveUp
MoveDown
MoveLeft
MoveRight
Activate
Back
```

平台 adapter 只生成 canonical key event。Host/key binding 层把 key event 映射为 FocusIntent。

Focus 核心不知道 Ctrl、Alt、keycode、Crossterm 或 winit。产品全局快捷键可以在进入 Focus 前处理。

## Focus Policy

每个 focus target 声明：

```text
FocusPolicy {
    focusable: bool,
    enabled: bool,
    visible: bool,
    captured_intents: IntentMask,
}
```

`enabled` 必须从 PEP 6 的 Availability 派生，不能形成第二份状态。`visible` 是 tree participation 属性，不属于交互 style state。

通常：

- Button capture `Activate`。
- Menu item capture `Activate`。
- 普通输入控件按自身能力 capture 编辑 intent；第一阶段不定义编辑 intent。
- 游戏 Canvas 可以 capture 四个 Move intent 和 Activate。
- 非交互 Text 和装饰叶子不可 focus。

hidden 或 disabled target 不得获得 focus，也不得收到 dispatch。

`captured_intents` 是声明数据，不是隐藏 callback。产品每帧可以根据状态重建 policy。

## Relation To Semantic Interaction State

Focus service 只拥有 PEP 6 的 Focus 轴：

```text
Unfocused | Focused
```

它不拥有 Selected、Active、Pressed、Checked、Expanded、Invalid 或 Busy。

`Activate` 只产生 Dispatch。产品 action 可以在成功后改变 Selected、Active、Checked 或 Expanded。Focus service 不根据控件类型猜测这些转换。

变为 Disabled 时，Focus reconciliation 移除 Focused。Selected、Active 和 Checked 等产品值保持不变。

## Structural Navigation

方向关系来自显式 UI 结构：

```text
FocusRow
FocusColumn
FocusGrid
ExplicitNeighbors
```

- Row 默认建立 Left/Right 邻居。
- Column 默认建立 Up/Down 邻居。
- Grid 使用逻辑 row/column 建立四向邻居。
- ExplicitNeighbors 覆盖单个方向。

布局矩形不参与 graph 构建。TUI 和 GUI 即使测量结果不同，也必须得到相同 graph。

`FocusNext/FocusPrevious` 使用 scope 内稳定声明顺序。Z 顺序和 paint source order 不改变遍历顺序。

实现可以直接在 registry 中保存邻居，不要求公开 `FocusGraph` 类型。

## Dispatch Algorithm

收到 intent 后按以下顺序处理。

### Step 1: Reconcile

先确保当前 focus 仍存在、visible、enabled 且位于 active scope。

### Step 2: Target Capture

当前 target capture 该 intent 时返回：

```text
Dispatch {
    target: FocusId,
    intent: FocusIntent,
}
```

Focus state 不自动变化。Host 把 `(target, intent)` 映射为产品 action。

这条规则让游戏 Canvas 可以消费方向键，而不需要 Focus 核心依赖游戏领域。

### Step 3: Navigation Fallback

未 capture 的导航 intent 按 graph 移动 focus：

- `FocusNext/Previous` 使用稳定遍历顺序，并在 scope 内循环。
- `Move*` 使用对应显式邻居。
- 邻居不可用时，沿同方向继续跳过 disabled/hidden target。
- 没有合法目标时返回 `Unhandled`，focus 不变。

### Step 4: Activate And Back Fallback

- 未 capture 的 `Activate` 返回 `Unhandled`。
- active nested scope 未 capture `Back` 时返回 `ExitScopeRequested(scope)`。
- root scope 未 capture `Back` 时返回 `Unhandled`，交给 host 处理退出或上级导航。

## Focus Outcome

状态机返回有限结果：

```text
FocusOutcome =
    FocusChanged { from, to }
  | Dispatch { target, intent }
  | ExitScopeRequested { scope }
  | Unhandled { intent }
```

Outcome 不直接包含产品领域对象。Host 或 product-view binding 把 target/intent 转换为产品 action。

一次 intent 最多产生一个主要 outcome。Focus change 后需要显示提示时，由渲染读取新 state，不增加第二个事件总线。

## Scope Model

Root UI 建立 root scope。Modal、弹出菜单或需要独占键盘的 Canvas 可以建立 nested scope。

Scope 声明：

```text
FocusScope {
    id,
    initial_focus: Optional<FocusId>,
    trap: bool,
}
```

进入 scope 时保存之前的 focused ID。选择 focus 的顺序：

1. scope 的有效 `initial_focus`。
2. scope 内声明顺序的第一个有效 target。
3. 没有 target 时保持 scope active，但 focus 为 None。

退出 scope 时：

1. 尝试恢复保存的 ID。
2. ID 不可用时执行父 scope reconciliation。

`trap = true` 时，Next/Previous 和方向导航不能离开 scope。Modal 必须使用 trap。

显示一个高 Z modal 不会自动进入 scope。产品必须同时声明 modal scope，避免视觉层和交互层隐式耦合。

## Rebuild Reconciliation

每帧 UI rebuild 后，Focus 使用新 registry 修复 state：

1. 当前 active scope 不存在时，逐级退回仍存在的父 scope。
2. 当前 focused target 仍有效时保留。
3. 否则尝试该 scope 的 initial focus。
4. 否则选择声明顺序的第一个有效 target。
5. 没有有效 target 时设为 None。

Reconciliation 不根据“离旧矩形最近”选择 target。

## Product Action Boundary

推荐产品 view 维护显式 binding：

```text
(FocusId, FocusIntent) -> ProductAction
```

Focus 核心只返回 Dispatch。Product core 只接收 ProductAction。两边都不依赖平台 key event。

Binding 可以是 match、静态表或产品 view 方法。Punctum 不建立字符串 action registry 或运行时 Service Locator。

## Focus Rendering

共享 UI 只暴露 `is_focused(FocusId)`。

Target theme 决定表现：

- Terminal 可以改变 style、前缀或边框字符。
- GUI 可以绘制 focus ring、边框或颜色。

Focus ring 不改变 layout。需要预留空间时，组件 border 必须在 focused/unfocused 状态下使用相同整数宽度。

## Failure Model

结构化错误至少包含：

```text
DuplicateFocusId
DuplicateScopeId
UnknownScope
TargetOutsideScope
InvalidInitialFocus
InvalidNeighbor
ScopeCycle
```

重复 ID、scope cycle 和跨 scope 非法引用应拒绝该 frame 的 focus registry。Target 消失和 disabled 属于正常 reconciliation，不是错误。

## Testing

### State Machine Tests

至少覆盖：

- 初始 focus。
- Next/Previous 循环。
- Row、Column、Grid 四向导航。
- explicit neighbor override。
- disabled/hidden skip。
- captured Move 产生 Dispatch，不移动 focus。
- 未 capture Move 使用 navigation fallback。
- modal enter、trap、Back 和 restore。
- rebuild 后 target 保留、消失和 scope 消失。

### Cross-Target Tests

同一 UI description 在 Terminal 和 GUI 上必须产生相同：

- registry 顺序。
- neighbor relation。
- intent outcome。
- 产品 action。

不能根据 target layout golden 推导 Focus 正确性。

### Product Fixtures

- Tetris：target game leaf capture 方向键；Pause modal trap focus；关闭后恢复游戏叶子。
- Game：HUD 菜单方向导航；日志不可 focus；Back 返回上级菜单。

## Backwards Compatibility

Focus API 是新增合同，没有稳定外部兼容承诺。迁移可以替换 host 中临时 menu index 和 modal focus 状态。

产品 action 和业务状态转换必须保持不变。键位变化不属于本 PEP，需由产品输入配置单独确认。

## Migration

1. 先在 `punctum-ui` 内实现纯 Focus module。
2. 用 Tetris 和 Game fixture 固定 ID、policy、scope 和 outcome。
3. Terminal 与 GPU host 使用同一 key binding 到 FocusIntent mapping。
4. 删除 host 中重复的菜单 index 和 modal focus 状态。
5. 达到独立复用门槛后再决定是否拆 `punctum-focus` crate。

## Rejected Alternatives

### Geometry-Based Direction Search

Terminal 与 GUI 的矩形不同。用像素距离选邻居会导致跨 target 行为分叉，也会把 layout 泄漏进 Focus。

### Full DOM Event System

当前没有 pointer、capture/bubble 和复杂文本输入需求。完整事件树会增加没有消费者的机制。

### Product Callbacks Inside Focus Core

回调会让纯状态转换难以测试，并可能反向依赖产品。结构化 Dispatch outcome 已足够。

### Z-Index Determines Focus

视觉覆盖和键盘导航是不同语义。Modal 通过 scope trap 建立交互边界。

### Separate TUI And GUI Focus State

这会让相同页面在不同 target 上产生不同 action。Focus 没有必要依赖 target。

### Immediate Focus Crate Split

PEP 定义的是行为合同，不是目录指标。没有稳定消费者前先放在 `punctum-ui` 内更容易修剪错误抽象。

## Open Issues

1. FocusNext/Previous 是否永久循环，还是允许 scope 配置 stop-at-edge。
2. FocusGrid 遇到不规则行时使用同列 clamp，还是要求显式 neighbor。
3. debug 工具是否需要输出 registry 的纯文本快照。
