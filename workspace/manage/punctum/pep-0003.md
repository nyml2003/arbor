## Status Note

PEP 7 将 GUI Stack、Terminal overlay 和 frame commit 分开验证。本文保留候选规则，但不再假定两端共享同一个层叠或合成合同。

## Abstract

本文定义 Punctum 的覆盖布局、绘制顺序、`z-index`、clip 和 frame ownership。

覆盖由 `Stack` 表达。每个 Stack 建立局部 stacking context。直接 child 使用局部 `ZIndex`，相同值按源顺序稳定绘制。后绘制的内容覆盖先绘制的内容。descendant 不能越过父 stacking context。

`z-index` 只决定绘制顺序，不自动创建离屏纹理或 compositing layer。Terminal target 把各层写入同一个最终 Terminal frame。GPU target 在同一个 acquired frame 中依次编码 Canvas、HUD、overlay 和 modal，最后只 submit/present 一次。

## Motivation

游戏页面至少包含以下层：

```text
game scene
HUD
menu or tooltip
modal
debug overlay
```

如果没有统一层叠合同，每个产品会分别处理覆盖、clip 和 present。常见后果包括：

- Canvas 自己获取并 present surface，HUD 无法继续绘制。
- 用一个全局巨大 `z-index` 解决局部覆盖，数值不断膨胀。
- GPU 为普通层叠创建离屏纹理，增加显存和带宽。
- Terminal 与 GUI 使用不同的 child 顺序。
- Focus 顺序错误地跟随视觉 Z 顺序。
- batching 改变透明内容的绘制结果。

需要把“谁先画”和“是否离屏合成”拆成两个问题。

## Goals

1. 定义覆盖节点和局部 Z 顺序。
2. 防止 descendant 逃出父 stacking context。
3. 保证 TUI 与 GUI 使用相同语义 paint order。
4. 保证一个平台 frame 只 acquire、submit 和 present 一次。
5. 允许 Canvas、HUD 和 modal 写入同一 frame。
6. 不为普通层叠引入离屏合成。

## Non-Goals

第一阶段不支持：

- CSS stacking context 全部规则。
- arbitrary transform 和 3D depth sorting。
- filter、blur、backdrop filter 和复杂 mask。
- group opacity。
- 自动 layer promotion。
- 离屏缓存和 retained compositing tree。
- pointer hit testing。

Focus 顺序由 PEP 4 定义，不由 Z 顺序推导。

## Terminology

### Paint Order

Paint order 是 target planner 必须保留的语义绘制顺序。

### Stacking Context

Stacking context 是 Z 顺序的局部边界。一个 context 内部的 child 可以排序。context 整体在父 context 中只占一个位置。

### ZIndex

`ZIndex` 是有符号 16 位 newtype。默认值为 0。负值允许出现在同一 Stack 的普通内容之前。

### Direct Composition

Direct composition 表示各层直接写入最终 Terminal frame 或当前 GPU render target，不创建中间纹理。

### Offscreen Composition

Offscreen composition 表示先把一组 child 画到中间 surface，再把结果作为一个整体混合到父 surface。

## Stack Node

`Stack` 是共享 UI 节点：

```text
Stack {
    fit: Loose | Expand,
    clip: None | Bounds,
    children: StackChild[],
}

StackChild {
    node,
    z_index: ZIndex,
    align: Alignment,
}
```

`Stack` 自身建立 stacking context。只有直接 `StackChild` 拥有 `z_index`。

Flex child 不接受 `z-index`。需要覆盖时，产品必须显式引入 Stack。这样布局流和绘制层叠不会互相污染。

## Stack Layout

Stack 使用 PEP 2 的 target-local constraints。

- `Expand`：有限轴使用父最大值；无界轴退化为 `Loose`。
- `Loose`：按所有 child preferred size 的逐轴最大值决定自身尺寸，再按父约束 clamp。
- 每个 child 在 Stack bounds 内按 alignment 放置。
- child overflow 不扩大一个已经 bounded 的 Stack。
- `clip: Bounds` 把 child paint clip 与 Stack bounds 相交。

Stack layout 不读取 `z_index`。改变 Z 顺序不能改变 measure 或 bounds。

## Stacking Algorithm

每个 Stack 按以下 key 对直接 child 稳定排序：

```text
(z_index ascending, source_index ascending)
```

绘制从小到大进行。后绘制的 child 位于视觉上方。

排序必须稳定。不得依赖 hash map iteration order。

示例：

```text
Stack
├─ z=0   CanvasView(game)
├─ z=10  HUD
├─ z=20  Popup
└─ z=30  Modal
```

这些值只在该 Stack 内有意义。嵌套 Stack 中的 `z=30000` 也不能覆盖父 Stack 的后续 sibling。

产品可以定义 `Scene/Hud/Overlay/Modal/Debug` 等本地常量。Punctum 不建立全局 Z 数值注册表。

## Stacking Path

Planner 为 paint item 建立逻辑 stacking path：

```text
[(ancestor_z, ancestor_source), ..., (local_z, local_source)]
```

该 path 用于调试、golden 和验证。实现可以在 tree traversal 中直接生成有序 command，不要求把 path 存入公开运行时对象。

Renderer 不得只按末级 `z_index` 做全局排序。那会允许 descendant 逃出父 context。

## Clip Stack

Clip 使用矩形交集：

```text
effective_clip = parent_clip intersect local_clip
```

空交集跳过该 subtree 的 paint planning。

Canvas、HUD 和 Text 都必须服从 effective clip。Canvas 可以在自己的 viewport 内增加更窄 scissor，但不能扩大祖先 clip。

第一阶段只要求 axis-aligned rectangle clip：

- Terminal target 转换为 cell rectangle。
- GPU target 转换为 scissor rectangle。

复杂 path clip 需要新的 PEP。

## Z-Index Is Not Compositing

普通 `z-index` 不创建中间 surface。

```text
layout tree
-> stable stacking traversal
-> ordered target commands
-> one final target
```

需要 offscreen composition 的典型功能包括 group opacity、filter、复杂 mask 和 subtree cache。第一阶段没有这些能力，因此不建立 `CompositingGroup`。

单个 primitive 的 alpha 不等于 group opacity。API 不得把两者使用同一个名称。

## Terminal Composition

Terminal target 先建立一个最终 `Surface<TerminalCell>`，再按 paint order 写入。

Terminal layer 需要区分：

- `Transparent`：不修改已有 cell。
- `Cell(value)`：覆盖当前位置，包括明确的空白 cell。

Terminal 没有通用半透明混合。请求半透明 primitive 时，target 必须使用明确 fallback 或返回 `UnsupportedCapability`，不能假装等同 GPU alpha。

所有层完成后：

```text
final Surface
-> diff previous Surface
-> Terminal patch plan
-> driver write
```

中间 layer 不单独 diff，也不直接写 Terminal。

## GPU Composition

GPU host 拥有 frame 生命周期：

```text
acquire surface texture once
-> begin frame
-> clear once
-> encode ordered paint commands
-> submit once
-> present once
```

Canvas renderer、GUI renderer 和 overlay renderer 接收当前 frame encoder、target view 和 effective clip。它们不得：

- 再次 acquire surface。
- 自行 present。
- 清除整个 target。
- 提交另一个破坏顺序的独立 frame。

普通矩形、图片和 glyph 可以使用标准 alpha blending，但必须保留 paint order。

## Batching Rule

Renderer 可以合并相邻且兼容的 command。Renderer 不得为了减少 draw call 跨越语义 paint barrier 重排 command。

以下变化至少构成 barrier：

- stacking context 顺序变化。
- effective clip 变化。
- target 或 viewport 变化。
- 与透明内容存在顺序相关的 overlap。

只有能够证明视觉等价时才能重排不透明、互不相交的 command。第一阶段默认不做这种证明。

## Frame Plan

Target planner 输出纯计划：

```text
FramePlan {
    clear,
    ordered_commands,
    diagnostics,
}
```

Planner 不持有 window、surface texture 或 Terminal session。Platform driver 执行计划。

计划中的资源使用类型安全 ID。不能把 wgpu handle、Android view 或 Terminal writer 放入共享 UI tree。

## Focus And Semantics

Z 顺序不改变：

- FocusNext/FocusPrevious 顺序。
- 方向导航关系。
- 可访问语义顺序。
- 产品 Action mapping。

视觉上位于顶部的 modal 必须显式建立 Focus scope trap。仅提高 `z-index` 不会自动抢占 focus。

隐藏 subtree 不参与 paint 和 focus。被其他内容遮挡但仍 visible 的节点不会自动从 focus graph 移除。

## Diagnostics And Failure

结构化 diagnostic 至少包含：

```text
InvalidStackBounds
ClipOverflow
UnsupportedCompositingEffect
FrameAlreadyAcquired
FrameAlreadyPresented
CanvasClearedParentTarget
```

重复 present、越权 clear 和 frame 生命周期错误应终止当前 frame，并由 host 恢复或重建 surface。普通内容被 clip 不是错误。

## Testing

### Pure Stacking Tests

至少覆盖：

- Z 正数、0 和负数。
- 相同 Z 的稳定源顺序。
- 嵌套 context 无法逃逸。
- Z 改变不影响 layout bounds。
- clip 交集和空 clip subtree skip。

### Terminal Tests

- Transparent 不覆盖底层。
- 明确空白覆盖底层。
- Canvas、HUD、modal 的最终 cell 顺序。
- 只对最终 frame 执行一次 diff。

### GPU Tests

- acquire、clear、submit、present 各一次。
- Canvas 和 HUD command 位于同一 FramePlan。
- scissor 遵守祖先 clip。
- renderer 不跨 barrier 重排透明 command。

### Cross-Target Tests

对双方共有的 Stack fixture，Terminal 与 GUI 应产生相同 stacking path 顺序。target-only leaf 和最终像素/Terminal cell 不参与一一比较。

## Backwards Compatibility

现有 GPU grid renderer 可以保留 logical command oracle，但其 acquire/clear/submit/present 所有权必须迁移到 host frame。该调整允许内部 breaking change。

Terminal 最终 `Surface` 和 patch 合同保持不变。新增透明 layer 语义不能改变明确空白 cell 的既有覆盖行为。

## Migration

1. 在 `punctum-ui` 中增加 Stack、StackChild 和 ZIndex。
2. 让 paint traversal 输出稳定的 stacking path。
3. 给 Terminal layer 增加 Transparent 与明确空白的区别。
4. 把 `punctum-wgpu` 的 surface acquire/present 从 grid renderer 提升到 frame owner。
5. 让现有 grid renderer 作为 Canvas command encoder 接入当前 frame。
6. 用 Tetris game + HUD + pause modal 验证顺序。

## Rejected Alternatives

### Global Z-Index

全局排序会破坏组件封装，并允许深层 child 覆盖不相关页面。局部 stacking context 可以限制影响范围。

### Child Order Only

只靠 child 顺序适合静态 Stack，但动态 HUD、popup 和 modal 经常需要保持语义 tree 顺序并调整视觉层级。局部 `ZIndex` 提供受控能力。

### One Offscreen Texture Per Layer

普通覆盖不需要离屏纹理。该方案增加显存、带宽和移动端 tile memory 压力。

### Depth Buffer For UI

UI 需要稳定 painter order、clip 和 alpha。把所有 UI 放进 depth buffer 会让透明排序和局部 context 更难控制。

### Canvas Owns Present

Canvas 自行 present 后，HUD 和 modal 无法进入同一 frame，也无法统一处理 surface loss。

## Open Issues

1. `ZIndex` 的公开范围是否限制在 `i16`，还是使用更小范围并提供 checked constructor。
2. debug overlay 是普通产品 Stack child，还是 host 注入的最终受控 layer。
3. 何种真实需求足以引入第一种 explicit offscreen compositing effect。
