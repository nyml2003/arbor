## Status Note

本文保留为语义词汇库存。PEP 7 不要求首个切片建立完整状态全集；每个状态轴只在真实组件需要且事实来源明确时进入合同。

## Abstract

本文定义 Punctum keyboard-first 交互元素的闭合语义状态集合。

状态由若干正交轴组成。`focused`、`selected`、`active` 和 `pressed` 不是同义词，也不应由同一个 service 管理。Focus service 只拥有 focus 轴。组件或产品状态拥有 selection、activation、check、expansion 等轴。输入 dispatcher 可以拥有短暂的 press 轴。

第一阶段 renderer 可以对部分状态使用相同外观，但不能改变状态含义。新增状态必须修改本 PEP，不能使用自由字符串扩张。

## Motivation

一个键盘控件可能同时：

- 已选中，但没有 focus。
- 有 focus，但尚未选中。
- 已激活某个模式，同时仍可获得或失去 focus。
- 正在处理任务，因此 busy。
- disabled，但保留 selected 值。

如果只提供 `normal/focused`，产品会把 selected、active 和 disabled 塞进临时 style flag。如果把所有状态放进 Focus service，Focus 又会开始管理产品值。

需要先定义完整词汇和所有权，再决定首版渲染多少外观。

## Scope Of The Universe

“全集”指 Punctum 当前 keyboard-first、无 pointer 范围内允许出现在公共组件合同中的全部交互状态轴。

本 PEP 不声称覆盖未来鼠标、触摸、拖放和文本编辑。引入 hover、drag、drop target、IME composing 等状态时必须新增或修改 PEP。

## Goals

1. 区分 focus、selection、activation 和 press。
2. 定义每个状态轴的唯一所有者。
3. 定义合法组合和默认状态。
4. 让 TUI 与 GUI 读取相同语义状态，但自由选择表现。
5. 允许首版暂不为所有状态设计独立视觉。
6. 阻止自由字符串和 target-specific flag 进入共享组件。

## Non-Goals

本文不定义：

- 完整 theme token 集合。
- CSS pseudo-class 或 selector cascade。
- pointer hover、drag 和 touch gesture 状态。
- 每个组件必须支持的所有状态轴。
- 产品领域状态的通用状态机。
- animation timeline。

## State Axes

### Availability

```text
Enabled | Disabled
```

`Disabled` 表示控件不能接收 focus、dispatch 或 activate。它不表示 hidden。

所有者：组件或产品 view，根据产品能力派生。

### Focus

```text
Unfocused | Focused
```

`Focused` 表示该元素是当前 keyboard intent target。

所有者：PEP 4 Focus state machine。

### Selection

```text
NotSelectable | Unselected | Selected
```

`Selected` 表示元素属于一个选择模型，并且当前被选择。选择可以在 focus 离开后保留。

例子：列表选择项、多选条目、当前选中的存档。

所有者：组件或产品 selection model。

### Activation

```text
NotActivatable | Inactive | Active
```

`Active` 表示元素关联的模式、工具或内容当前正在生效。它是持久语义，不是按键按下瞬间。

例子：当前生效的工具、已经打开的面板、正在接收游戏控制的模式。

所有者：组件或产品状态。

收到 `Activate` intent 不会自动把元素设为 Active。产品 action 成功改变状态后，下一个 view 才反映 Active。

### Press

```text
Released | Pressed
```

`Pressed` 是 key-down 到 key-up 之间的短暂反馈。它不等于 Active。

所有者：input/interaction dispatcher。

首版如果 canonical input 不能可靠提供 down/up 配对，可以只实现 Released。不得用 Active 假装 Pressed。

### Check

```text
NotCheckable | Unchecked | Checked | Mixed
```

Check 表示 checkbox、toggle 或三态值。它不等于列表 Selected。

所有者：组件或产品值模型。

### Expansion

```text
NotExpandable | Collapsed | Expanded
```

用于 tree node、disclosure 和 submenu。

所有者：组件或产品状态。

### Editability

```text
NotEditable | Editable | ReadOnly
```

第一阶段不实现文本编辑器，但先保留该语义，避免以后把 ReadOnly 混入 Disabled。ReadOnly 内容仍可 focus 或复制；Disabled 不能交互。

所有者：组件或产品能力。

### Validity

```text
NotValidated | Valid | Invalid
```

用于已经参加校验的输入或选择。Invalid 不自动 Disabled。

所有者：产品 validation model。

### Activity

```text
Idle | Busy
```

Busy 表示对应 action 正在执行。Busy 是否同时 Disabled 由组件合同决定，不能由 theme 猜测。

所有者：application/product state。

## Excluded Properties

以下属性不是交互 style state：

- `Visible/Hidden`：决定节点是否参加 layout、paint 和 focus registry。
- `ZIndex`：决定局部绘制顺序。
- `FocusScope`：决定导航边界。
- `Primary/Danger/Quiet`：属于组件 visual variant。
- 产品的 HP、游戏 phase、网络状态：属于产品领域数据。

这些概念不能作为自由 flag 混入 interaction state set。

## Compact Representation

本 PEP 定义语义轴，不强制运行时为每个节点保存一个大 struct。

实现可以使用闭合 enum/bitset 表示非默认状态：

```text
Disabled
Focused
Selected
Active
Pressed
Checked
Mixed
Expanded
ReadOnly
Invalid
Busy
```

组件类型负责说明哪些轴适用。`NotSelectable/NotCheckable/NotExpandable` 等“不适用”值可以由组件类型表达，不要求占用运行时 bit。

禁止：

- `HashSet<String>` 保存状态。
- target 私自增加 `terminal_blinking` 或 `gpu_hovered` 到共享集合。
- 同时设置 `Checked` 和 `Mixed`。
- 从 style 颜色反向推断语义状态。

## Source Of Truth

同一状态只能有一个事实来源：

| Axis | Source of truth |
| --- | --- |
| Availability | component/product view |
| Focus | Focus service |
| Selection | selection/product state |
| Activation | component/product state |
| Press | interaction dispatcher |
| Check | component/product value |
| Expansion | component/product state |
| Editability | component/product capability |
| Validity | validation state |
| Activity | application/product state |

UI description 在构建时把这些来源合成为只读 semantic state。Theme 和 renderer 只能读取，不能修改。

## Legal Combinations

大多数轴正交，可以组合。例如：

```text
Focused + Selected
Focused + Active
Selected + Disabled
Invalid + Focused
Busy + Active
```

以下组合非法或需要规范化：

- `Disabled + Focused`：Focus reconciliation 必须移除 Focused。
- `Disabled + Pressed`：dispatcher 必须清除 Pressed。
- `Checked + Mixed`：构造错误。
- 同一轴多个互斥值：构造错误。

`Disabled + Selected/Active/Checked` 合法。Disabled 只限制交互，不抹掉产品值。

## Relation To Focus Policy

PEP 4 的 `enabled` 与本 PEP Availability 是同一事实，不能分别保存后再同步。

推荐：

```text
FocusPolicy.enabled = (Availability == Enabled)
```

`Focused` 只能从 Focus state 派生，产品 view 不得自行设置。

Canvas capture、modal scope 和 neighbor graph 不属于 semantic state set。

## Style Resolution

组件先根据 semantic state 选择语义 style token，target theme 再把 token 解析为 Terminal style 或 GUI paint。

```text
component variant + semantic state
-> semantic style recipe
-> target theme
-> target paint
```

Punctum 不定义 CSS 式 selector specificity。组件 recipe 必须明确处理组合，不依赖字符串规则覆盖顺序。

Target 可以暂时让多个状态共享外观。例如 Terminal 可以让 Active 与 Selected 使用同一种强调色。但它们在 semantic state、测试和 action 逻辑中仍然不同。

## Integer Style Rule

状态 style 遵守 PEP 2：所有 geometry 数值使用整数。

- focus border width 是整数。
- pressed offset 是整数。
- alpha 使用 `u8`。
- 不允许 `0.5px` focus ring 或小数缩放。

第一阶段，交互状态变化不得改变组件外部 layout metrics。Focused/Selected/Active 可以改变颜色、字形、边框内容，但应预留相同整数边框宽度，避免键盘导航导致布局跳动。

## TUI And GUI Rendering

共享的是状态语义，不是表现。

Terminal 可以使用：

- foreground/background style。
- 整数宽度边框。
- 前缀 glyph。
- reverse 或 underline 等明确 capability。

GUI 可以使用：

- 整数宽度 focus ring。
- fill、border、glyph 或图标变化。
- `u8` alpha primitive。

Target 不支持某种表现时可以使用较简单表现。Target 不返回“Unsupported Selected State”；状态存在，外观可以退化。

## Transition Rules

### Focus

Focus state machine 改变 Focused。产品 action 不直接写 Focused。

### Activate

`Activate` dispatch 只是请求。产品处理成功后可以改变 Selected、Active、Checked、Expanded 或其他领域值。Focus service 不猜测结果。

### Press

可靠 key-down 设置 Pressed。对应 key-up、focus loss、disable 或 scope exit 清除 Pressed。

### Disable

变为 Disabled 时：

- Focus service 在 reconciliation 中移除 Focused。
- dispatcher 清除 Pressed。
- Selected、Active、Checked 等产品值保持不变，除非产品规则另有动作。

## Testing

### Vocabulary Tests

- 公共状态只能来自闭合集合。
- 互斥状态不能共存。
- target-only style 不扩张 semantic state。

### Ownership Tests

- Focus service 只改变 Focused。
- Activate outcome 不自动改变 Active/Selected/Checked。
- Disabled reconciliation 清除 Focused/Pressed。
- 产品 rebuild 能保留 Selected/Active/Checked。

### Cross-Target Tests

同一共享产品流程在 TUI 与 GUI 上比较 semantic state，不比较颜色、glyph 或 border 像素。

### Layout Stability Tests

组件在 Supported interaction state 组合之间切换时，外部 `LayoutSize` 保持相同。

## Backwards Compatibility

语义状态 API 是新增合同。现有临时 bool 可以 breaking rename，但必须先映射到本文状态轴，避免 `active` 同时表示 Focused、Pressed 和 Selected。

状态拆分不得改变产品领域值。未实现独立视觉的 target 可以暂时保持原外观。

## Migration

1. 盘点现有组件中的 bool 和 style flag。
2. 把 `focused/selected/active/pressed` 分开命名。
3. 规定每个状态的事实来源。
4. 先实现 Enabled/Disabled、Focused、Selected、Active。
5. Pressed、Checked、Expanded、ReadOnly、Invalid、Busy 按真实组件需求接入。
6. 未实现独立外观的状态仍保留 semantic fixture。

## Rejected Alternatives

### Normal/Hover/Active Enum

该模型来自 pointer CSS，无法同时表达 Focused + Selected + Disabled 等组合，也混淆 Active 与 Pressed。

### Focus Service Owns All States

Selected、Checked 和 Active 是产品或组件值。Focus service 管理它们会越过领域边界。

### Free-Form State Strings

字符串允许每个 target 和组件扩张词汇，无法检查合法组合和所有权。

### Require Every Component To Store Every Axis

Text、Button、Checkbox 和 Canvas 的适用状态不同。全集是词汇边界，不是大而全的存储结构。

### Ignore Unrendered States

首版没有独立视觉不代表状态不存在。否则后续补样式时会改变产品语义和测试合同。

## Open Issues

1. `Pressed` 是否等 canonical input 增加可靠 key-up 后再实现。
2. `Busy` 是否需要区分 determinate progress；当前不进入交互状态全集。
3. 组件 style recipe 使用 match 还是静态状态表；先由真实 Button/Menu fixture 决定。
