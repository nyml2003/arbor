Author: nyml

## Status Note

PEP 7 将 Flex 和整数 GUI 布局退回 Game-owned probe。本文保留算法候选，但在真实 GUI 与第二消费者提供证据前不进入实施或公共 API。

## Abstract

本文定义 Punctum 第一阶段的共享布局合同。

TUI 与 GUI 共享同一个约束布局算法。每次布局只运行在当前 target 的本地单位中。Terminal 使用行列单位。GUI 使用整数虚拟像素。两者不共享最终矩形，也不比较数值相等。

所有进入 Punctum layout 或 target style 的 geometry 数值都必须是整数。无法整除的空间按稳定源顺序分配余数。平台 scale factor 可以是小数，但只能在 layout 到 framebuffer 的最终 adapter 边界使用。

第一阶段支持 `Flex`、`Padding`、`Align` 和固定尺寸叶子。Flex 支持单行 Row/Column、gap、basis、grow、shrink 和主轴、交叉轴对齐。算法使用确定性的整数分配。

## Motivation

“支持 FlexBox 子集”不足以指导实现。以下问题如果不先固定，会分别泄漏到组件、TUI backend 和 GPU backend：

- `Auto` basis 在什么约束下测量。
- 空间不足时是 shrink、clip 还是报错。
- Terminal 的列和 GUI 像素能否使用同一种 geometry 类型。
- 整数除法余数分给哪个 child。
- Text 在 basis 探测和最终宽度下是否重复 shaping。
- overflow 是 panic、错误还是可渲染诊断。

布局必须先成为纯、确定、可测试的服务，target renderer 才能保持简单。

## Goals

1. 固定 target-local 坐标和约束类型。
2. 固定叶子 measure 与最终 layout 的调用合同。
3. 固定单行 Flex 的 grow、shrink、对齐和舍入算法。
4. 保证同一输入在同一 target metrics 下得到相同结果。
5. 允许 Terminal 与 GUI 复用算法，但使用不同测量结果。
6. 把 overflow 表达为有界输出和结构化诊断。
7. 禁止小数 style 和小数 layout 进入共享 UI。

## Non-Goals

第一阶段不支持：

- wrap 和多行 Flex。
- percentage、viewport unit 和 calc。
- order。
- baseline alignment。
- min-content、max-content 和 CSS intrinsic sizing 完整语义。
- margin collapse。
- absolute positioning。
- 从一个 target 的布局缓存复用另一个 target 的结果。
- 浮点亚像素布局。

层叠、`z-index` 和绘制顺序由 PEP 3 定义。

## Terminology

### Layout Unit

`LayoutUnit` 是有符号整数。实现至少覆盖 32 位范围。

Terminal target 的 X 单位是终端列，Y 单位是终端行。GUI target 的 X/Y 单位是整数虚拟像素。`LayoutUnit` 的数值没有跨 target 含义。

### Constraints

约束包含每个轴的最小值和最大值：

```text
Constraints {
    min_width,
    max_width,
    min_height,
    max_height,
}
```

最大值可以是 `Unbounded`。最小值必须有限且非负。有限最大值不得小于最小值。

### Measure

Measure 回答节点在给定约束下需要的尺寸。结果必须落在约束内。

### Layout Artifact

叶子最终布局可以同时产生 target-local artifact，例如 glyph layout、Terminal grapheme layout 或 Canvas viewport。Paint 必须复用该 artifact，不能用不同规则重新测量。

## Coordinate Types

共享 UI 使用独立类型：

```text
LayoutPos
LayoutSize
LayoutRect
LayoutInsets
Constraints
```

以下转换必须显式发生：

```text
world/grid -> Canvas internal projection
layout      -> target paint plan
layout      -> physical framebuffer pixels
```

禁止：

- 用 `GridRect` 保存 GUI 或 Terminal widget bounds。
- 用 `PixelRect` 参与 Flex 计算。
- 把 Terminal layout cache 传给 GUI target。
- 假设一个 Terminal layout unit 是方形。

## Integer Style Contract

共享 geometry style 只使用整数或整数 newtype：

- width、height、min、max。
- padding、gap、border width、corner radius。
- `grow/shrink` 权重。
- `z-index`。

Target-only 数值 style 也只能使用整数。例如 GUI font size、line height、letter spacing 使用整数虚拟像素；primitive alpha 使用 `u8`。Terminal 不需要定义 GUI font size 或 alpha 字段。

不支持 `0.5px`、`1.25rem`、浮点 grow 和百分比 style。

三个等权 child 分配 10 个单位时，结果必须是 `4, 3, 3`。余数从源顺序靠前且仍有资格接收空间的 child 开始分配。

Window DPI 或移动端 device scale 可以是小数。Platform adapter 在 layout 完成后把整数虚拟像素映射为物理像素，并使用明确的 round policy。该 scale factor 不能进入 Flex 算法、共享 style、Node identity 或 layout cache key。

## Node Contract

节点至少支持两个阶段：

```text
probe(node, constraints, metrics) -> PreferredSize
place(node, exact_or_bounded_constraints, metrics) -> PlacedNode
```

`probe` 用于取得 Flex basis。`place` 使用分配后的最终约束，并生成 paint 可复用的 layout artifact。

同一个节点、约束和 metrics 的结果必须稳定。布局服务可以缓存相同请求。

叶子不得在 measure/place 中：

- 修改产品状态。
- 获取或 present GPU surface。
- 写 Terminal。
- 读取未显式传入的全局 DPI、字体或主题。

## Supported Nodes

### Fixed Leaf

固定叶子声明 preferred size，并按父约束 clamp。

### Padding

Padding 先从约束中扣除 insets，再布局 child，最后把 insets 加回结果。扣除和加回必须使用 checked 或 saturating arithmetic，并产生 overflow diagnostic。

### Align

Align 不改变 child 的 preferred size。父节点分配额外空间后，Align 在该空间内按 X/Y 对齐 child。

### Flex

Flex 是单行容器：

```text
Flex {
    direction: Row | Column,
    gap: LayoutUnit,
    main_align: Start | Center | End | SpaceBetween,
    cross_align: Start | Center | End | Stretch,
    children: FlexItem[],
}
```

每个 item 包含：

```text
FlexItem {
    basis: Auto | Fixed(LayoutUnit),
    grow: u16,
    shrink: u16,
    min_main: LayoutUnit,
    max_main: Finite(LayoutUnit) | Unbounded,
    align_self: Inherit | Start | Center | End | Stretch,
}
```

`grow = 0` 表示不接收正向剩余空间。`shrink = 0` 表示空间不足时不收缩。

## Flex Algorithm

### Step 1: Resolve Container Space

布局服务确定主轴和交叉轴，并解析 target theme 提供的 gap。负 gap 非法。

`n` 个 child 的固定 gap 总量为：

```text
max(n - 1, 0) * gap
```

运算溢出产生 `ArithmeticOverflow`，并使用可表示的有界值继续生成诊断结果。

### Step 2: Probe Basis

每个 item 取得初始 basis：

- `Fixed(v)` 使用 `v`。
- `Auto` 调用 child `probe`。

`Auto` probe 的主轴最小值为 0。主轴最大值使用容器可用最大值；容器无界时传入 `Unbounded`。交叉轴沿用容器交叉约束。

probe 结果按 item 的 `min_main/max_main` clamp。

### Step 3: Compute Free Space

当容器主轴最大值有限时：

```text
free = available_main - gaps - sum(basis)
```

当主轴无界时，不执行 grow。容器主轴尺寸由 basis、gap 和父约束决定。

### Step 4: Grow

`free > 0` 时，按 `grow` 权重分配空间。

达到 `max_main` 的 item 被 freeze。剩余空间重新在未 freeze item 中分配，直到空间耗尽或没有可 grow item。

### Step 5: Shrink

`free < 0` 时，按以下权重收缩：

```text
shrink_weight = shrink * max(basis, 1)
```

达到 `min_main` 的 item 被 freeze。剩余负空间重新分配，直到不足消失或没有可 shrink item。

如果所有 item 到达最小值后仍放不下，保留最小尺寸并产生 `MainAxisOverflow`。父节点的 clip 决定超出部分是否可见。

### Step 6: Integer Remainder

所有分配使用整数。

先按权重计算整数商。未分配的正负余数按源顺序逐个分给仍未 freeze 的 item。相同输入不能依赖 hash iteration order。

### Step 7: Final Place

每个 child 使用最终主轴尺寸调用 `place`。

主轴约束为 exact。交叉轴：

- `Stretch` 且父交叉轴有限时使用父分配的 exact 尺寸。
- 其他情况使用父交叉约束，并按 child 最终尺寸对齐。

Text、Canvas 和 target-only leaf 可以在此阶段生成最终 layout artifact。GUI Image 属于 target-only leaf。basis probe artifact 只有在约束完全相同时才能复用。

### Step 8: Align

分配结束后，`main_align` 只处理未被 grow 使用的正向剩余空间。

- `Start`：剩余空间放在末尾。
- `Center`：前后分配，奇数余数放在末尾。
- `End`：剩余空间放在开头。
- `SpaceBetween`：只增加 child 之间的间距。少于两个 child 时等同 `Start`。

`cross_align` 或 `align_self` 决定 child 在交叉轴的位置。

## Theme Metrics

共享 node 保存语义 spacing token，不保存跨平台绝对尺寸。

Target theme 在布局开始前把 token 解析为 `LayoutUnit`：

```text
Space::Xs
Space::Sm
Space::Md
Space::Lg
```

第一阶段 token 集合保持很小。自定义固定值只允许出现在产品 target theme 或 Canvas 内部，不能让 Terminal 列数泄漏进共享组件。

## Diagnostics And Failure

布局返回有界结果和 diagnostics：

```text
LayoutDiagnostic =
    InvalidConstraints
  | NegativeSpacing
  | ArithmeticOverflow
  | MainAxisOverflow
  | CrossAxisOverflow
  | LeafMeasureFailure
```

无效约束和负 spacing 是构造错误，正常路径应拒绝生成 frame。

内容 overflow 可以生成 frame。Host 决定在开发模式记录诊断。Renderer 不得因普通窗口缩小而 panic。

## Target Responsibilities

Terminal target 提供：

- grapheme 和 terminal-width 测量。
- 行列 spacing metrics。
- Terminal GridView preferred size。

GUI target 提供：

- font shaping 和 glyph metrics。
- 虚拟像素 spacing metrics。
- Image 和 GPU Canvas preferred size。

Target 不得复制 Flex 分配算法。

## Testing

### Algorithm Tests

至少覆盖：

- Row 和 Column 对称性。
- 0、1、多个 child 的 gap。
- grow 权重和 max freeze。
- shrink 权重和 min freeze。
- 正负整数余数的稳定顺序。
- bounded 和 unbounded 主轴。
- Stretch 和 align_self override。
- overflow diagnostic 与有界结果。

### Property Tests

对有效输入检查：

- child 主轴尺寸不低于 min_main。
- 有限 max_main 不被超过。
- 无 overflow 时，child、gap 和外部剩余空间之和等于容器主轴尺寸。
- 输出顺序与输入顺序一致。
- 相同输入产生相同输出。

### Cross-Target Tests

双方共有的容器 fixture 在 Terminal 和 GUI 上只比较结构不变量。不能断言 tree 一一对应或矩形数值相等。

## Backwards Compatibility

Punctum 当前未建立稳定外部 UI API。本 PEP 允许 `punctum-ui` 的 geometry 和 layout API 发生 breaking change。

迁移不得改变 `punctum-grid` 的离散 geometry 语义，也不得把 Terminal grapheme width 行为移入共享 Flex 算法。

## Migration

1. 在 `punctum-ui` 中新增 `Layout*` 和 `Constraints`，停止使用 `Grid*` 表达 UI geometry。
2. 用本 PEP 算法替换当前临时 Row/Column 逻辑。
3. 让 Terminal fixture 和 GUI fixture 分别提供 metrics 与 leaf measurer。
4. 增加 Tetris HUD 和 Game 菜单 fixture。
5. 合同稳定后再公开 API。

## Rejected Alternatives

### Full CSS FlexBox

CSS FlexBox 包含 wrap、baseline、intrinsic sizing、percentage 和大量兼容规则。第一阶段没有对应产品需求。

### Terminal Cell As Universal Layout Unit

Terminal Cell 不是 GUI 像素，也不是游戏 tile。复用该类型会把宽字符和非方形比例泄漏到 GUI。

### Float Layout

浮点会增加跨平台舍入差异。Punctum 第一阶段使用整数虚拟像素和明确余数规则。

### Fractional Style With Integer Output

先接受小数 style、最后再取整，会让舍入发生位置依赖实现顺序，也会污染 cache key。共享 style 从输入阶段就拒绝小数。

### Grow Without Shrink

只支持 grow 会让小窗口和移动端 viewport 直接溢出。最小 shrink 是响应式布局的必要能力。

### Re-Measure During Paint

Paint 重新测量会导致布局矩形与 glyph、图片或 Canvas 实际绘制不一致。

## Open Issues

1. `LayoutUnit` 使用 `i32` newtype，还是带 checked constructor 的专用 scalar。
2. 语义 spacing token 是否需要 `None` 和 `Xl`。
3. 开发模式如何展示 overflow diagnostic。

这些问题不改变算法主体，可以通过实现 fixture 决定。
