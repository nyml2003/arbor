## Status Note

PEP 7 接受“GUI Canvas 与 Terminal Grid 是不同实现路径”这一问题判断，但 custom-leaf API 仍需真实页面证明。本文暂不构成公共 trait、payload 或 frame 合同。

## Abstract

本文定义游戏内容如何嵌入 Punctum 页面。

共享 UI 只提供最小 custom-leaf 外部合同：measure、placed bounds、clip 和 Focus policy。Terminal 使用自己的 `GridView` 叶子。GUI 使用自己的 `CanvasView` 叶子。两者不共享 payload、内部坐标、paint command 或资源类型。

产品可以复用页面 shell、Focus ID 和 Action 语义，但可以分别构造 Terminal view 与 GUI view。Punctum 不要求两个 target 的 tree 节点一一对应。

## Motivation

游戏世界和 HUD 有不同职责：

- 页面 layout 决定游戏区域占多大、位于哪里。
- 游戏 renderer 决定 tile、sprite、camera 和 atlas 如何投影。
- HUD、menu 和 modal 必须继续在游戏区域上方绘制。
- Terminal 需要 grapheme/cell projection。
- GPU 需要 viewport、scissor 和 draw commands。

把这些差异压入通用 Cell 或通用 Canvas payload，会形成新的万能类型。反过来，让游戏 renderer 自己管理 window/frame，又会破坏 HUD 合成。

需要共享的是外部边界，不是内部画法。

## Evidence For The Boundary

当前至少有两个真实投影：

- Tetris board 在 Terminal 中把一个逻辑 tile 投影为多个终端列。
- 同一个 board 在 GPU 中把一个逻辑 tile 投影为一个方形 quad。

Game map 也需要 Terminal grid 和 GPU scene 两种输出。

这些消费者都需要 bounds、clip 和 keyboard focus，但不需要相同 payload 或命令。因此 custom-leaf 外部合同成立；通用 Canvas 内部模型不成立。

## Goals

1. 让游戏区域参与 PEP 2 布局。
2. 让游戏区域服从 PEP 3 stacking 和 clip。
3. 让游戏区域参与 PEP 4 Focus capture。
4. 允许 Terminal GridView 与 GUI CanvasView 保留不同内部模型。
5. 保证 Canvas/Grid renderer 不拥有平台 frame lifecycle。
6. 避免运行时 leaf registry、万能 command enum 和跨 target payload。

## Non-Goals

第一阶段不定义：

- 通用 scene graph。
- 通用 sprite、tile、camera 或 atlas 类型。
- TUI 图片协议。
- GPU API wrapper。
- 游戏 physics 坐标。
- target 间相同截图或相同 layout rect。
- 第三方插件的动态 leaf ABI。

## Architectural Role

Custom leaf 是布局树中的终点：

```text
Page container
└─ Stack
   ├─ target game leaf
   ├─ HUD
   └─ Modal
```

共享 layout 只观察：

```text
preferred size
final bounds
effective clip
focus metadata
```

layout 不观察 leaf payload。leaf 不观察 sibling 或父容器实现。

## Target-Specific Leaves

### Terminal GridView

Terminal product view 构造 Terminal-owned `GridView`。它可以保存：

- 产品只读 grid snapshot。
- tile 到 terminal cell 的投影策略。
- Terminal style token。
- 必要的 resource ID。

GridView planner 输出带透明语义的 Terminal cell layer，并写入当前最终 Terminal surface。

一个 `TerminalCell` 永远表示一个终端列。一个逻辑 tile 占几列由 GridView projection 决定，不能使用 presenter 全局 `cell_width`。

### GUI CanvasView

GUI product view 构造 GUI-owned `CanvasView`。它可以保存：

- 产品只读 scene/grid snapshot。
- camera/view state。
- sprite、texture、atlas 等 GUI resource ID。
- GUI renderer 需要的 target-local参数。

CanvasView planner 输出当前 GPU target 能理解的 commands。它不转换成 TerminalCell，也不进入 shared UI crate 的通用 enum。

## No Shared Payload

Punctum 不定义 `CanvasPayload`、`AnyCanvasCommand` 或跨 target `GpuCell`。

推荐数据流：

```text
Product snapshot
├─ terminal product-view -> GridView payload -> Terminal planner
└─ GUI product-view      -> CanvasView payload -> GPU planner
```

产品可以抽取双方真正共享的只读 board/map projection model。该模型属于产品或独立纯领域 crate，不属于 UI framework。

如果 Rust 实现需要泛型，泛型只作为编译期承载方式。PEP 不要求公开 `Node<C>`、trait object 或 type-erased registry。先按实际调用点选择最小实现。

## Shared Page Composition

产品可以复用不依赖 target leaf 的 builder：

```text
common_hud(snapshot)
common_pause_menu(snapshot)
common_focus_bindings(snapshot)
```

然后分别构造：

```text
terminal_page = page(GridView(...), terminal_hud(...))
gui_page      = page(CanvasView(...), gui_hud(...), gui_images(...))
```

复用不是指标。GUI avatar、图片背景或动画没有 TUI 对应物时，直接留在 GUI view。Terminal 不创建 placeholder node 来保持 tree 对称。

## Layout Contract

Custom leaf 实现 PEP 2 的 probe/place 行为。

第一阶段 sizing policy 保持最小：

```text
Preferred { width: integer, height: integer }
Expand
```

- `Preferred` 按父约束 clamp。
- `Expand` 在有限轴填满父分配；无界轴必须提供 target preferred fallback。
- 所有外部 bounds 使用整数 LayoutUnit。
- 三等分等剩余空间处理由父 Flex 完成，leaf 不自行使用小数补偿。

Canvas 内部 world 或 physics 是否使用浮点由产品决定。它不能反向改变共享 UI style 和 layout 的整数合同。

## Clip And Viewport

Custom leaf 接收 PEP 3 已计算的 effective clip。

Terminal GridView：

- 把 clip 转换为终端 cell rectangle。
- 不写入 bounds/clip 外的 cell。
- 宽 grapheme 在右边界放不下时使用 Terminal 既有 clip/fallback 规则。

GUI CanvasView：

- 把 bounds 转换为 viewport。
- 把 effective clip 转换为 scissor。
- 内部 camera 只影响 Canvas 内容，不影响 HUD layout。

leaf 可以缩小 clip，不能扩大祖先 clip。

## Focus Contract

GridView 与 CanvasView 可以使用相同语义 `FocusId`，但它们是两个 target view 中的不同节点。

游戏区域常用 policy：

```text
focusable = true
captured = MoveUp | MoveDown | MoveLeft | MoveRight | Activate
```

PEP 4 返回 `Dispatch { target, intent }`。产品把 intent 转成游戏 action。

Custom leaf 不直接读取平台 key event，也不修改 Focus state。Pause modal 出现时，由 modal scope trap 接管 focus。

## Frame Ownership

Custom leaf 只向当前 target plan 编码内容。

Terminal：

```text
GridView layer
-> shared final Terminal surface
-> one diff/patch
-> one driver flush
```

GPU：

```text
host acquires frame
-> CanvasView encodes game commands
-> HUD encodes UI commands
-> Modal encodes overlay commands
-> host submits/presents once
```

GridView/CanvasView 不得 acquire、clear whole parent target、flush、submit 或 present。

## Resource Ownership

共享 UI tree 不保存平台 resource handle。

- Product snapshot 保存领域数据和稳定资源身份。
- Target resource service 把身份解析为 glyph、texture、buffer 或 Terminal style。
- Platform driver 保存真实 handle。

资源缺失由 target planner 返回结构化 diagnostic。Terminal 不需要认识 GUI 图片或 texture identity。

## Failure Model

至少区分：

```text
InvalidLeafBounds
MissingPreferredSizeForUnboundedAxis
ProjectionOverflow
MissingTargetResource
UnsupportedTargetCommand
LeafWroteOutsideClip
LeafOwnedFrameLifecycle
```

产品 snapshot 无效属于产品错误。Platform surface loss 属于 host/runtime 错误。不要把所有失败压成 `CanvasError(String)`。

## Testing

### Shared Boundary Tests

- custom leaf 按父约束得到整数 bounds。
- leaf 不改变 sibling layout。
- effective clip 只缩小不扩大。
- Z 顺序中 leaf、HUD、modal 保持 PEP 3 合同。
- focus capture 产生相同产品 action。

### Terminal GridView Tests

- logical tile 到多列 Terminal cell 的投影。
- CJK/continuation 与右边界 clip。
- transparent 与明确空白覆盖。
- 不写 bounds 外 cell。

### GUI CanvasView Tests

- viewport/scissor 来自 placed bounds 和 clip。
- game、HUD、modal 共用一个 FramePlan。
- 缺失 resource 返回结构化 diagnostic。
- Canvas renderer 不 acquire/present。

### Product Tests

Tetris 和 Game 分别维护 Terminal/GUI tree fixture。跨 target 只比较共享 Focus/Action 和产品状态转换，不比较 leaf payload。

## Backwards Compatibility

Punctum 当前没有稳定 custom-leaf API。本 PEP 允许移除 `SurfaceView`、通用 Canvas payload 和 presenter 全局 `cell_width`。

迁移必须保持 Terminal Unicode/continuation 行为、grid diff/apply 合同和 GPU grid logical oracle。Tetris 与 Game 的产品规则不变。

## Migration

1. 把现有 Terminal game surface 明确命名为 GridView 或 Terminal game leaf。
2. 把现有 `punctum-gpu` grid renderer 收窄为 GUI Canvas 的一种 renderer。
3. 移除 shared `Image` 和通用 Canvas payload 设想。
4. 把 GPU surface ownership 提升到 host frame。
5. 分别接入 Tetris Terminal page 和 GUI page。
6. 只有出现第二种相同 custom leaf 接口后，才抽取公开 trait。

## Rejected Alternatives

### Shared CanvasView Payload

Terminal 需要 cell projection，GUI 需要 camera/resource/draw commands。共享 payload 会变成两端字段的并集。

### Universal Cell

Cell 无法合理表达 GUI glyph shaping、图片、viewport 和 scene commands。

### Terminal Image Fallback In Shared UI

Terminal 没有图片需求时，不应为了 GUI Image 增加 fallback、协议和错误分支。

### Runtime Leaf Registry

按字符串或 type ID 查找 renderer 会隐藏依赖，并让节点在 composition root 之外获得 service。编译期显式依赖更简单。

### Canvas Owns A Child Window Or Surface

独立 surface 会破坏 HUD/Modal 的统一 Z 顺序和一次 present 合同。

### Identical Cross-Target Tree

这会迫使 GUI-only Image 和 Terminal-only projection 互相伪装。应复用共同 builder，而不是追求节点一一对应。

## Open Issues

1. 第一版实现是否只使用产品 leaf enum，完全不增加 custom-leaf trait。
2. GridView/CanvasView 的 preferred fallback 应由 product view 还是 target theme 提供。
3. target resource service 达到什么规模后才拆独立 crate。
