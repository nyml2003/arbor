# 注释规则

这份文档定义当前仓库的注释规范。

目标：

- 注释解释边界、动机和非直观点
- 不把显而易见代码翻译成自然语言
- 不为了“看起来完整”而堆长注释

## 1. 什么值得写注释

- 公共类型的职责和边界
- 核心流程函数为什么这么分层
- 某段实现为什么不能更直接地写
- 容易读错的规则细节
- 临时设计取舍和退出条件

## 2. 什么不该写注释

- 变量赋值的表面描述
- 逐行翻译代码
- 已经能从函数名直接看出的行为
- 过时的历史背景

## 3. 注释形式

- 类型、核心边界函数：
  - 允许用块注释 `/** ... */`
- 局部复杂逻辑：
  - 允许用短块注释 `/* ... */`
- 普通小函数：
  - 默认不写注释，除非存在明显误读风险

## 4. 风格规则

- 优先写“为什么”，其次写“边界”，最后才写“做了什么”
- 注释用短句，避免长段背景故事
- 不把 TODO 写成长期常驻说明
- 如果代码已经改掉，注释必须同步改掉

## 5. 当前仓库的执行规则

- 现有高密度块注释文件：
  - 不要求一次性清洗
- 新增代码：
  - 不要继续无节制复制旧风格
- 后续拆分文件时：
  - 顺手把注释收敛成更少、更准的版本

## 6. 判断标准

一条注释如果删掉后：

- 代码仍然完全一样好懂

那这条注释大概率不该存在。
