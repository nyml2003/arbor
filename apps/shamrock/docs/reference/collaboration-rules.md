# Shamrock 协作规则

## 当前协作模式

这次任务里，已经稳定下来的协作模式是：

- 先锁目标和验收标准，再开始实现
- 实现按小步推进，不一次性写很多文件
- 每完成一段，就更新 `docs/current/status.md`
- 每次停下时，先回读进度文档，再判断是否到当前 milestone
- milestone 判断要明确写出来，不靠默认理解
- 测试和覆盖率是硬约束，不是最后补的附加项

## 可复用规则

### Incremental Delivery With Progress Checkpoint

- Pattern: 多步实现任务里，如果没有一开始就锁定执行节奏，后面容易在“大步改动”和“补流程约束”之间来回回调。
- Trigger Signals:
  - 用户在实现开始后补充“不要一次性写很多文件”这类节奏要求。
  - 用户要求每次停止时都更新文档，并用文档判断是否到 milestone。
  - 任务有明确验收点，比如 demo、覆盖率、阶段性可运行结果。
- Recommended Workflow:
  1. 开始实现前，先锁目标、验收标准、覆盖率门槛和当前 milestone。
  2. 同时锁执行节奏：是否小步写文件，是否每次停下都写文档，是否需要回读文档再做 milestone 判断。
  3. 每完成一个子阶段，就更新 `docs/current/status.md`，写清当前状态、当前问题和当前判断。
  4. 停止前必须回读进度文档，并明确说明“现在是否已经到当前 milestone”。
  5. 只有当前 milestone 达成后，才进入下一层复杂度。
- Scope: 多阶段实现、原型开发、从空仓库推进到 demo 的任务。
- Skip When:
  - 任务是单文件小修。
  - 用户明确要求一次性完成，不需要阶段性汇报。
