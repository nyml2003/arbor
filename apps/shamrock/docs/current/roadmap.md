# Shamrock 迭代计划

## 1. 当前范围

这份计划只保留当前仍未完成的主线。

## 2. 格式、AI 与更强外壳

目标：

- 在核心、view 和 replay 都稳定后，继续扩格式能力、AI 能力和更强外壳

## 3. 下一阶段优先顺序

1. 补完整的数据模型
2. 更完整的单打格式约束
3. 规则层继续去 demo 化
4. 可替换 AI 策略
5. 双打
6. 搜索接口和脚本接口
7. 更强的观战视图 / 脚本视图

## 4. 当前建议执行顺序

### Step 1：补数据模型

优先做：

- 继续把当前 species stats 的语义从“demo 直接战斗数值”收敛成真正的数据层模型
- 在现有可选 IV / EV / level 之上继续补 `Nature`
- 再补更完整最终能力值公式

当前状态：

- `Nature` 已补上
- demo 内容已切到表驱动
- 当前下一步更值得做的是格式层，而不是继续堆数据字段

### Step 2：补格式层

优先做：

- 更完整的合法动作约束
- 更明确的模式规则
- 为后续双打预留更稳定的格式边界

### Step 3：继续收敛规则层

优先做：

- 把当前仍留在 core 里的 demo 规则继续外移
- 收敛能力值计算、状态结算、目标逻辑的边界

### Step 4：抽 AI 策略边界

优先做：

- 把 `simple-rule-ai` 从 CLI demo 运行壳层里继续抽离
- 让 battle loop 能挂可替换策略，而不是只绑定一个 demo AI

### Step 5：更强外壳

优先做：

- 观战视图
- 脚本视图
- 更明确的 replay 浏览/检查入口

### Step 6：扩更高复杂度玩法

优先做：

- 双打
- 搜索接口
- 更复杂 AI

## 5. 当前约束

继续推进 Phase 5 时，仍然要守住这些约束：

- 纯函数式结算入口
- 状态只通过 `BattleOp` 改动
- replay 可重放、可恢复、可校验
- `battle-view` 不反向污染核心
- `battle-mechanics` 继续保持纯规则计算边界
- 新增实现不要继续放大超长文件

## 6. 当前风险

- `battle-core` / `battle-data` / `battle-cli main` 仍然偏大
- 更完整格式和更强 AI 一旦直接堆进现有壳层，会重新破坏刚刚收敛的边界
- 双打如果没有先补格式与 view 抽象，后续代价会明显变高
