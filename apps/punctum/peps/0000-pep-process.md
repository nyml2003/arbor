# PEP 0000：PEP 流程

- Status: Active
- Type: Process
- Created: 2026-07-10

## 摘要

PEP 用于记录影响 Punctum 公共模型、模块边界、兼容性或长期开发方式的决策。小修复、局部实现细节和可以随时撤销的实验不需要 PEP。

## 状态

- `Draft`：问题和方案仍在修改。
- `Provisional`：方向已选，但必须通过列出的实验或实现验证。
- `Accepted`：决策已被验证，后续实现应遵守。
- `Active`：持续生效的流程类提案。
- `Rejected`：方案未采用，文档保留原因。
- `Superseded`：已被另一份 PEP 替代。

## 类型

- `Standards Track`：公共 API、架构、数据合同或兼容规则。
- `Process`：开发、评审和发布流程。
- `Informational`：重要背景或使用指导，不规定实现。

## 必需内容

标准类 PEP 至少包含：

1. 问题和目标。
2. 明确的非目标。
3. 决策及其边界。
4. 被拒绝的方案和原因。
5. 风险。
6. 验证方法和接受条件。

## 生命周期

1. 新建编号文件，状态设为 `Draft`。
2. 在索引中登记。
3. 对尚未被原型证明的架构决策使用 `Provisional`。
4. 完成验证后，根据证据改为 `Accepted` 或 `Rejected`。
5. 改变已接受的公共合同，应新建 PEP，并在旧 PEP 中标记 `Superseded`。

PEP 记录决策，不代替测试。实现与 PEP 不一致时，应修正实现或更新提案状态，不能让两者长期分叉。
