# TEP-0004: 响应式协议

## 本质

手动版的 SolidJS Signal 响应式系统。SolidJS 用 Proxy 自动追踪 signal → component 的依赖关系。我们做显式订阅：组件 mount 时声明依赖，signal 变更时标记组件脏，事件循环末尾批量渲染。

放弃自动追踪的原因：TUI 单线程同步渲染，每秒最多 60 帧。自动追踪的复杂度（Proxy、effect 栈、嵌套追踪）在这个约束下收益为零。显式订阅的订阅列表可直接调试，signal 变更影响范围确定。

## 数据流

单向。和 React 的 `props down, events up` 一致：

```
用户交互 → 组件回调 → 业务层写状态 → 标记脏组件 → 批渲染
```

组件只读状态，不写状态。类型系统保证：组件持只读句柄，无写方法。

## 受控 vs 非受控

和 React 的受控/非受控组件区分一致：

- **受控组件**：值由外部状态驱动。外部改值 → 组件重渲染。如 Text 的 text 和 style 字段。
- **非受控组件**：状态在组件内部。组件通知外部发生了什么，但外部不直接控制内部状态。如 Input 的光标位置和未提交编辑文本——这些是编辑过程的临时状态，on_submit 时才把最终值交给外部。

框架两种都支持。受控走 ReadSignal，非受控走内部字段 + 回调。

## 批处理

事件循环末尾统一渲染，不在每次状态变更时立即渲染。和 React 的 batch update 机制目的一致：一次事件处理中多个 setState 合并为一次渲染。

不需要显式 batch API——事件循环结构天然提供批处理。高频推送（日志流、监控指标）全部合并到一帧。帧率上限 16ms，防止高频推送占满 CPU。

## 生命周期

组件 mount → 订阅依赖的状态值。事件循环中状态变更 → 标记脏。事件循环末尾 → 统一渲染。组件 unmount → 退订。Drop 兜底退订。

和 React 的 `componentDidMount → shouldComponentUpdate → componentWillUnmount` 语义一致，但更简单——没有 shouldComponentUpdate（Signal 的相等比较自动跳过无变化更新），没有 getDerivedStateFromProps（单向数据流不需要）。

## 已决议

- 显式订阅，不做自动追踪
- 受控和非受控两种模式都支持
- 批处理，事件循环末尾渲染
- 帧率上限 60fps（防止高频推送，不是目标帧率）
