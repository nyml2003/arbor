# TEP-0007: 终态分层与测试策略

## 本质

当前代码比旧文档更新。后续实现以代码事实和本 TEP 为准，不再为了兼容旧 TEP 里的过期说法保留旧接口。

这个项目仍处在 v0 孵化阶段。终态合理优先于兼容旧 API、旧测试和旧模块名。旧测试如果锁定了错误结构或错误行为，直接改掉。

## 分层

当前分层是五层：

| 层 | 职责 | 不允许做的事 |
| --- | --- | --- |
| domain | `arbor-tui-domain`。纯类型、布局计算、渲染模型、组件树协议、事件、动作类型、响应式状态和后端端口 | 读写真实终端、轮询真实输入、管理真实时间 |
| application | `arbor-tui-application`。运行时状态转换。输入当前状态和事件，输出渲染意图和副作用意图 | 直接写终端、直接依赖 crossterm、依赖 adapters |
| adapters | `arbor-tui-adapters`。真实终端、模拟终端、输入源、ANSI emit | 写业务规则和组件状态机 |
| widgets | `arbor-tui-widgets`。内置组件和 builder | 拥有运行时主状态、依赖 application/adapters |
| testing | `arbor-tui-testing`。E2E driver、输入脚本、屏幕断言、样式断言、输出断言 | 依赖真实终端 |

旧 crate 名不再保留兼容层。`primitives`、`reactive`、`render`、`widget` 已并入 `domain`。旧 `backend` 已改为 `adapters`。旧 `arbor-tui` 主 crate 已改为 `application`。

## 运行时状态

运行时状态只能通过明确入口改变。入口接收输入，返回结果：

```text
AppState + RuntimeInput -> RuntimeStepResult
```

`RuntimeInput` 包含键盘事件、resize、force render 等框架输入。`RuntimeStepResult` 说明是否需要渲染、是否需要清屏、是否退出。

事件循环只负责：

1. 从 adapter 读取输入。
2. 调用 runtime step。
3. 按 step 结果执行 render/emit。

焦点、dirty、resize、退出状态不能散落在多个入口里随手改。

## 命名

命名必须表达语义，不表达历史实现：

- 布局容器叫 `Stack` / `StackWidget`，由 `Col` 和 `Row` builder 创建。
- `BoxWidget` 这个名字不再使用。它描述的是历史实现，不是领域概念。
- 分配 widget id 的对象叫 factory，不叫 manager。它只创建节点，不管理运行时状态。
- 测试输入和测试终端使用 testing/adapters 语境命名，不和生产后端混在一起。

## 测试策略

TUI 的 E2E 测试必须 mock 输入和输出。测试从用户输入脚本开始，到最终屏幕和输出结束。

核心测试资产：

- `TuiTestDriver`：创建 app、root tree、scripted input、test terminal。
- `AnsiTuiTestDriver`：使用测试专用 ANSI replay terminal。它先走 crossterm ANSI 编码，再把输出回放到测试屏幕。
- 输入脚本：字符、方向键、Tab、Enter、Escape、resize。
- 屏幕断言：文本、单元格、前景色、背景色、属性、宽字符续列。
- 输出断言：dirty region 数量、render 次数、终端输出是否发生、真实 adapter 是否强制写入颜色。

当前 `arbor-tui-testing/tests` 有 80 个 runtime E2E 场景。其中 77 个使用模拟输入和模拟后端，3 个使用 ANSI replay terminal。ANSI replay 场景覆盖 light theme 输入尾部重绘、嵌套 input 行背景和复杂 dashboard 颜色。它们用于发现 xterm.js 这类终端解释层的问题。

真实输出层也必须测。`arbor-tui-adapters` 要验证 crossterm 输出对每个可见 cell 写入 foreground、background 和 attrs。partial dirty region 不能用行尾 clear 让终端默认色参与渲染。

局部 widget 测试只验证组件自己的静态渲染和局部行为。跨组件行为、焦点、事件冒泡、dirty、resize 和 emit 必须走 E2E。

## 注释

注释只写四类内容：

- 不变量。
- 边界。
- 终端限制。
- 测试意图。

不要解释一眼能看懂的赋值、循环和转发。中文注释用短句，直接写原因和结果。

## 已决议

- 终态合理优先，不保旧 API。
- 旧文档偏移时，以本 TEP 和代码事实为准。
- runtime step 是状态修改入口。
- E2E 测试必须覆盖输入、状态、渲染、样式和输出。
- `BoxWidget` 改名为 `StackWidget`。
- 五层 crate 边界已落地，不保留旧 crate 名兼容层。
