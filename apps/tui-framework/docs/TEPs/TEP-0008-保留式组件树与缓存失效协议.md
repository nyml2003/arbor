# TEP-0008: 保留式组件树与缓存失效协议

## 本质

arbor-tui 需要从“每帧重建整棵 widget tree”的立即模式，演进为“带 key 的保留式 widget tree”。

目标不是改变终端屏幕 diff 协议。TEP-0002 里的 VirtualScreen 逐行 diff 仍然成立。本 TEP 增加的是 widget tree 层的 reconciliation、layout cache、render cache 和 dirty rect 协议。

新的运行时模型：

```text
state -> view() -> NodeSpec tree
      -> reconcile old WidgetTree
      -> collect DirtyKind
      -> reuse valid layout/render cache
      -> render dirty subtrees or viewport
      -> diff dirty rects
      -> emit ANSI
```

## 背景

当前 facade 每帧都会执行：

```text
view() -> build_root() -> new WidgetNode tree
```

这样有三个问题：

1. widget 实例不稳定。widget 内部缓存跨帧留不住。
2. `DirtyTracker` 只知道哪个 widget 脏了，不知道是 render、layout 还是 structure 脏。
3. 运行时即使知道 dirty widget，仍然要全树 measure、layout、render，再整屏 diff。

上一轮性能优化已经证明局部优化有效：

- ScrollView viewport render 显著降低长内容渲染成本。
- diff row-slice fast path 能降低屏幕比较成本。
- focus map 按需重建有效，但动态 child 场景需要明确 structure dirty。

下一阶段性能上限取决于 retained tree 和 cache invalidation，而不是继续给单个 widget 加局部补丁。

## 目标

1. widget 实例可以跨帧复用。
2. framework 能区分 render/layout/structure/theme/full 五类失效。
3. layout cache 有明确输入和失效规则。
4. render cache 有明确输入和失效规则。
5. DirtyTracker 可以安全缩小 render 和 diff 范围。
6. 所有缓存都可以先用 shadow mode 验证命中率和正确性。

## 非目标

1. 不重写 TEP-0002 的 VirtualScreen diff 协议。
2. 不在第一阶段替换所有 Widget API。
3. 不立刻实现完整局部 render。
4. 不为了兼容旧 facade API 保留错误结构。项目仍处于 v0，终态正确优先。
5. 不做自动依赖追踪。Signal 仍然采用 TEP-0004 的显式订阅模型。

## 核心类型

新增稳定 identity：

```rust
pub struct WidgetKey(String);

pub enum NodeIdentity {
    Keyed(WidgetKey),
    Path(Vec<u16>),
}
```

新增 dirty 分级：

```rust
pub enum DirtyKind {
    Render,
    Layout,
    Structure,
    Theme,
    Full,
}
```

新增 widget revision：

```rust
pub struct WidgetRevision {
    props: u64,
    layout: u64,
    render: u64,
    structure: u64,
}
```

revision 不要求对外暴露。它是 runtime 判断 cache 是否可复用的内部版本号。

## NodeSpec 与 WidgetNode

facade 的 `view()` 不应该直接决定 retained widget 实例。

需要拆成两层：

| 类型 | 职责 |
| --- | --- |
| `NodeSpec` | 用户 view 产物。描述希望出现什么组件、props、children、key。 |
| `WidgetNode` | runtime retained 实例。保存 widget 状态、订阅、cache、revision。 |

`NodeSpec` 是可丢弃描述。`WidgetNode` 是可复用实例。

builder API 增加可选 key：

```rust
Text::new("title").key("title")
Input::new().key("prompt")
TranscriptMessage::new(...).key(format!("message:{id}"))
```

无显式 key 时，runtime 使用结构路径兜底：

```text
root/0/2/1
```

兜底 path 只能保证局部稳定。列表、聊天记录、可重排项目必须使用显式 key。

## Reconciliation

reconcile 输入：

```rust
pub fn reconcile(
    old: Option<WidgetNode>,
    next: NodeSpec,
    ctx: &mut ReconcileCtx,
) -> WidgetNode;
```

复用条件：

```text
same identity
same widget type
compatible state model
```

替换条件：

```text
key 不同
type 不同
structure 不兼容
debug/test 下发现同级 duplicate key
```

复用时：

1. 保留 widget 内部状态。
2. 更新 props。
3. 比较 props revision。
4. 递归 reconcile children。
5. 输出 DirtyKind。

替换时：

1. 对旧节点执行 unmount。
2. 创建新 widget。
3. mount 新节点。
4. 标记 `DirtyKind::Structure`。
5. 标记 focus map 失效。

## DirtyKind 语义

DirtyKind 合并规则：

```text
Full > Theme > Structure > Layout > Render
```

各级含义：

| DirtyKind | 触发 | 影响 |
| --- | --- | --- |
| Render | 光标、选中态、scroll offset、loading spinner | 只清 render cache |
| Layout | 文本尺寸、wrap 宽度、padding/margin/flex 改变 | 清当前节点 layout，并向祖先传播 |
| Structure | children 增删、Tabs active child 变化、key/type 替换 | 清子树 cache，重建 focus map |
| Theme | theme revision 变化 | 清全树 render cache，可复用结构 |
| Full | resize、root fallback rebuild、backend reset | 全量 measure/layout/render |

`DirtyTracker` 改为：

```rust
pub struct DirtyTracker {
    dirty: HashMap<WidgetId, DirtyKind>,
    full: bool,
}
```

Signal 更新默认标记 `Render`。组件可以声明更高等级：

```rust
fn dirty_on_signal(&self, signal: SignalId) -> DirtyKind;
```

例子：

- `ScrollView.scroll_y`：Render。
- `Text.text` 且 wrap none：Render 或 Layout，取决于宽度和高度是否可能变化。
- `RichText.lines`：Layout。
- `Tabs.active`：Structure。
- `Theme`：Theme。
- `Resize`：Full。

## Layout Cache

layout cache 分 measure 和 layout 两层。

```rust
pub struct MeasureCacheEntry {
    available: Size,
    props_hash: u64,
    children_measure_hash: u64,
    output: SizeConstraint,
}

pub struct LayoutCacheEntry {
    parent_rect: Rect,
    props_hash: u64,
    children_measure_hash: u64,
    output: WidgetLayoutInfo,
}
```

命中条件：

```text
available 相同
props_hash 相同
children_measure_hash 相同
widget layout revision 未变
```

失效规则：

1. `Render` 不清 layout cache。
2. `Layout` 清当前节点 measure/layout cache。
3. 子节点 measure 变化时，父节点 layout cache 失效。
4. 父节点 rect 变化时，所有依赖父尺寸的子节点 layout cache 失效。
5. `Structure` 清当前子树 layout cache。
6. `Full` 清全部 layout cache。

第一阶段必须使用 shadow mode：

```text
仍然执行 full measure/layout
同时计算 cache hit/miss
比较 cache output 与 full output
只记录统计，不改变渲染结果
```

shadow mode 通过后，才能启用 layout cache。

## Render Cache

render cache 保存 widget 输出片段。

```rust
pub struct RenderCacheEntry {
    rect: Rect,
    theme_rev: u64,
    focus_state: FocusState,
    render_rev: u64,
    screen: VirtualScreen,
}
```

命中条件：

```text
WidgetId 相同
rect 相同
theme_rev 相同
focus_state 相同
render_rev 相同
```

失效规则：

1. 当前节点 `Render`：清自己 render cache。
2. 当前节点 `Layout`：清自己 render cache。
3. rect 变化：清自己 render cache。
4. focus 变化：清 old focus 和 new focus render cache。
5. theme 变化：清全树 render cache。
6. `Structure`：清替换子树 render cache。

第一批只给 leaf widget 启用 render cache：

- Text
- RichText
- Input
- Button
- Divider

容器 widget 后做。ScrollView 单独处理，只缓存 viewport 或可见行片段，不缓存完整 `content_h` 大屏。

## Dirty Rect Diff

局部 diff 必须排在 retained tree、layout cache 和 dirty kind 之后。

原因：只知道 dirty widget id 不够。layout 变化会移动兄弟节点。widget 变小会留下旧区域。ancestor clipping 会限制可见区域。

dirty rect 结构：

```rust
pub struct DirtyRect {
    old: Option<Rect>,
    new: Option<Rect>,
    clip: Rect,
}
```

生成流程：

```text
dirty widget ids
-> lookup old layout rect
-> lookup new layout rect
-> expand affected sibling/ancestor rects when layout dirty
-> apply ancestor clip
-> merge rects
-> render affected subtrees
-> diff only affected rows/rects
```

diff API：

```rust
pub fn diff_regions(
    old: &VirtualScreen,
    new: &VirtualScreen,
    regions: &[Rect],
) -> Vec<DirtyRegion>;
```

必须包含 old rect。否则 widget 变小后旧内容不会被清理。

## Focus 协议

focus map 依赖可见 widget tree。

以下情况必须重建 focus map：

1. root 初次 mount。
2. `DirtyKind::Structure`。
3. key/type 替换。
4. Tabs active child 变化。
5. widget focusable 状态变化。
6. resize 导致可见结构变化。

普通 `Render` 不应重建 focus map。

focus 切换只标记 old focus 和 new focus 为 `Render` dirty。

## Crate 边界

建议模块归属：

| 模块 | crate | 职责 |
| --- | --- | --- |
| `WidgetKey`, `NodeIdentity`, `DirtyKind` | `arbor-tui-domain` | 纯类型 |
| `RetainedTree`, `ReconcileCtx` | `arbor-tui-domain` | 树复用和失效结果 |
| layout/render cache entry | `arbor-tui-domain` | 纯缓存数据结构 |
| runtime cache owner | `arbor-tui-application` | 持有跨帧 cache |
| facade `.key(...)` API | `arbor-tui` | 用户入口 |
| benchmark counters | `aster-tui` / bench-log | 水位分析 |

`widgets` 不拥有全局 cache。widget 可以暴露 revision 和 dirty hint，但 cache 生命周期由 application runtime 管。

## 实施阶段

### 阶段 1：identity 和 key

新增 `WidgetKey`、`NodeIdentity` 和 `.key(...)` API。

验收：

- explicit key 能进入 NodeSpec。
- 无 key 时 path identity 稳定。
- 同级 duplicate key 在测试中报错。
- 现有渲染行为不变。

### 阶段 2：reconciliation

新增 `NodeSpec -> WidgetNode` reconcile。

验收：

- key 相同、type 相同复用 widget 实例。
- key 相同、type 不同替换实例。
- unmount/mount 顺序正确。
- Tabs active child 切换后 focus 正确。

### 阶段 3：DirtyKind

替换粗粒度 dirty set。

验收：

- Signal 更新能产生正确 dirty kind。
- DirtyKind 合并顺序正确。
- Render dirty 不触发 layout dirty。
- Structure dirty 触发 focus rebuild。

### 阶段 4：layout cache shadow mode

记录 cache hit/miss，但不改变输出。

验收：

- shadow output 与 full layout output 完全一致。
- benchmark 日志包含 layout cache 命中率。
- 文本变长导致兄弟节点下移不漏算。

### 阶段 5：启用 layout cache

在 shadow mode 通过后启用。

验收：

- 全测试通过。
- 标准基准 P95 不退化。
- layout stage 有可测下降。

### 阶段 6：render cache shadow mode

先覆盖 leaf widget。

验收：

- cached fragment 与 fresh render 完全一致。
- focus/theme/rect 变化全部能失效。

### 阶段 7：启用 render cache

先 leaf，后容器。

验收：

- 全测试通过。
- 标准基准 P95/P99 不退化。
- streaming、scroll、palette 场景 render stage 下降。

### 阶段 8：dirty rect diff

基于 old/new layout rect 限制 diff 范围。

验收：

- widget 变小后旧区域被清理。
- sibling 被 layout 推移后不残留。
- ancestor clipping 正确。
- ANSI replay 测试颜色不残留。

## 测试要求

必须新增专项测试：

1. duplicate key 报错。
2. key 相同类型相同复用实例。
3. key 相同类型不同替换实例。
4. 无 key 的 path identity 在静态树中稳定。
5. 列表重排时 explicit key 保留实例。
6. Tabs active child 切换后 focus 正确。
7. 文本变长导致兄弟节点下移不会漏画。
8. widget 变小后 old rect 被清理。
9. resize 后全量失效。
10. theme 切换后颜色不残留。
11. ScrollView 只渲染 viewport。
12. focus old/new 节点都重绘。

每阶段必须跑：

```bash
cargo test --workspace
cargo test -p aster-tui --features bench-log
cargo run -p aster-tui --features bench-log -- --bench
```

## 性能指标

标准基准继续使用 aster `headless_strict`。

最低要求：

- P95 不退化。
- P99 不退化超过 5%。
- C 水位帧保持 0。
- cache shadow mode 无输出差异。

目标：

- layout stage 在静态树和流式 token 场景下降。
- render stage 在 scroll、streaming 场景下降。
- diff stage 在局部更新场景下降。
- S 水位帧占比保持 95% 以上。

## 风险

1. key 设计不稳定会导致状态错复用。
2. layout cache 失效不完整会导致漏画或错位。
3. render cache 漏掉 focus/theme/rect 会导致样式残留。
4. dirty rect 不包含 old rect 会导致旧内容残留。
5. path identity 只能兜底，不能用于可重排列表。

## 已决议

- key 用于 widget tree reconciliation，不用于 VirtualScreen diff。
- NodeSpec 是 view 描述，WidgetNode 是 retained 实例。
- DirtyKind 分为 Render、Layout、Structure、Theme、Full。
- layout cache 必须先 shadow mode。
- render cache 先 leaf widget，再扩展容器。
- dirty rect diff 必须包含 old rect 和 new rect。
- ScrollView 不缓存完整 content_h 大屏。
- `widgets` 不拥有全局 cache，cache 生命周期属于 application runtime。
