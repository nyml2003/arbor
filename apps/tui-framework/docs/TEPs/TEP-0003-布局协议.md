# TEP-0003: 布局协议

## 本质

CSS Flexbox 的终端等价物。同样的盒模型（margin/padding/content）、同样的两级协商（measure 报约束 → layout 分配空间）、同样的弹性分配算法。

差异：终端是等宽字符网格，不是像素画布。排列方向只有行和列。尺寸是整数。没有嵌套滚动上下文。这些约束让布局引擎可以手写而不是引入 taffy——核心算法不到 200 行。

## 两级协商

**measure（子→父，自底向上）**：父传入可用空间，子返回尺寸约束。叶子组件测量内容（文本宽度、行数），容器组件聚合子项。

**layout（父→子，自顶向下）**：父根据 flex 规则把空间分给子项。子获得坐标和内容区尺寸。递归向下。

两级协商方向不可逆。这和 CSS 的 measure/layout 完全一致。

## Flexbox 子集

支持的：排列方向、主轴对齐（前/中/后/均分）、交叉轴对齐（前/中/后/撑满）、弹性比例、固定尺寸、内外间距。

不支持的：
- wrap（换行）— 终端屏窄，嵌套容器可替代多行
- gap（间隙）— margin + padding 已覆盖
- align-self（子项独立交叉轴对齐）— 嵌套容器可替代
- 百分比尺寸 — 弹性比例 + SIGWINCH 全量重排已覆盖
- min/max 约束 — 弹性分配已覆盖大多数场景

## 盒模型

和 CSS 一致：外边距（兄弟间距）→ 内边距（内容偏移）→ 内容区（渲染区域）。固定尺寸定义的是内容区，不含内外边距。子组件可用空间 = 父内容区 - 子外边距。

## SIGWINCH

终端尺寸变化 → 全树重排。不做增量——全屏尺寸都变了，增量重排省不了什么但引入脏布局状态管理的复杂度远超收益。

## 视口裁切

ScrollView 的子组件可能远大于视口。裁切只在渲染阶段发生，不在布局阶段——和 CSS overflow 的"布局不裁、渲染裁"一致。这使 List/Table 可以做虚拟滚动——视口外零开销。

## 已决议

- Flexbox 子集：手写，不用 taffy
- 两趟协商方向不可逆
- 弹性余数轮询分配
- 空容器尺寸为零
- 视口裁切只在 render 阶段
