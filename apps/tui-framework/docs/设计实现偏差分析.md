# 设计实现偏差分析

TEP-0001 ~ TEP-0006 定义的四条协议与三层层架构，对比 `crates/arbor-tui-*` 的实际代码，逐项列出偏差。

生成时间: 2026-07-05
最后更新: 2026-07-05 (4 项严重 + 4 项中等偏差已修复，剩余仅轻微偏差)

---

## 一、总览

四条协议的核心机制均已实现：VirtualScreen + diff（渲染协议）、measure/layout 两趟协商（布局协议）、Signal/ReadSignal + DirtyTracker（响应式协议）、WidgetNode 枚举 + FocusManager + 事件冒泡（组件协议）。三层 crate 拆分（domain / infra / app）符合 TEP-0006 设计，依赖方向正确。

以下按 TEP 和严重程度列出具体偏差。

---

## 二、严重偏差（设计意图矛盾或功能断裂）

### 2.1 渲染管线每帧无条件全量执行（TEP-0004）✅ 已修复

**TEP 设计**：事件循环末尾统一渲染，只在有脏组件时才执行管线。

**代码现状**：`event_loop.rs:run_event_loop()` 每轮 while 循环无条件调用 `app.render_widget_tree()`，不检查 `dirty_tracker.is_empty()`。

```rust
// event_loop.rs:66 — 每轮循环都渲染，不管有没有脏组件
while app.is_running() {
    let events = poll_events(input);
    // ... handle events ...
    app.render_widget_tree(root, theme, backend);  // 无条件执行
}
```

`app.rs:render_widget_tree()` 内部在 `frame_seq > 0` 且 `regions.is_empty()` 时才跳过 emit，但仍执行了完整的 measure → layout → render → diff 管线。无脏组件时这 4 步是纯浪费。

**影响**：空闲时 CPU 不接近零——每 100ms（poll_timeout 超时）执行一次完整的布局+渲染管线。违背 TEP-0001 的"空闲 CPU 接近零"硬约束。

**修复方向**：渲染前检查 `dirty_tracker.is_empty()`，空则直接 return false。或者检查事件队列是否有新事件 + 是否有脏组件，两者皆无才 skip。

---

### 2.2 组件 mount 时不自动订阅 Signal（TEP-0004）✅ 已修复

**TEP 设计**：组件 mount → 订阅依赖的状态值。组件 unmount → 退订。Drop 兜底退订。

**代码现状**：`mount_tree()` 调用每个 widget 的 `on_mount()`，但所有内置组件的 `on_mount()` 都是默认空实现（来自 `#[derive(Default)]` 的 Widget trait）。没有任何代码在 mount 时调用 `signal.subscribe(widget_id)`。

```rust
// widget.rs — Widget trait 默认实现
fn on_mount(&mut self) {}     // 空
fn on_unmount(&mut self) {}   // 空
```

Drop 实现倒是做了退订：
```rust
// widget.rs:279-283
impl Drop for TextWidget {
    fn drop(&mut self) {
        self.text.unsubscribe(self.id);
        self.style.unsubscribe(self.id);
    }
}
```

订阅退订不对称——退订有，订阅没有。目前只能由应用层手动调用 `signal.subscribe(widget_id)`，这不是 TEP 设计的自动生命周期管理。

**影响**：组件持有 ReadSignal 但未订阅，signal.set() 不会标记该组件为脏，状态变更不触发重渲染。受控组件模式不工作。

**修复方向**：在每个内置组件的 `on_mount()` 中，遍历其持有的 ReadSignal 并调用 `subscribe(self.id)`。或者提供 `Widget::subscriptions()` 方法返回信号列表，由 `mount_tree()` 统一订阅。

---

### 2.3 TabsWidget::on_key 破坏响应式链（TEP-0004 / TEP-0005）✅ 已修复

**TEP 设计**：ReadSignal 是只读视图，由外部 Signal 驱动。

**代码现状**：`widget.rs:367-390` 中 `TabsWidget::on_key()` 直接覆盖 `self.active`：

```rust
// widget.rs:377 — 用常量覆盖 ReadSignal，断开与源 Signal 的连接
Key::ArrowRight | Key::Char('l') => {
    let next = (old + 1) % self.tabs.len().max(1);
    self.active = ReadSignal::constant(next);  // BUG: 断开响应式链
}
```

`self.active` 字段类型是 `ReadSignal<usize>`。当用户按左右箭头切 tab 时，这行代码用一个新的 `ReadSignal::constant()` 替换了原来的 ReadSignal。原来的 Signal 及其订阅关系全部断开。外部通过 `on_switch` 回调通知了变更，但 `self.active` 和源 Signal 之间不再同步。

**影响**：外部通过 `tabs_signal.set(2, &mut dirty)` 切 tab 无效——TabsWidget 持有的已经是另一个 ReadSignal 了。这是响应式数据流断裂。

**修复方向**：TabsWidget 不应直接持有 `active: ReadSignal<usize>` 并尝试修改它。要么：
- 通过 `on_switch` 回调让外部改 Signal（保持 ReadSignal 只读语义）
- 或使用内部 `active_index: usize` + 外部 `active: ReadSignal<usize>` 双字段

---

### 2.4 Tabs::children() 返回空切片（TEP-0005）✅ 已修复

**TEP 设计**：组件树遍历覆盖所有子组件，焦点遍历进入 tab 内容。

**代码现状**：`focus.rs:232` 中 `WidgetNode::children()` 对 Tabs 变体返回空：

```rust
// focus.rs:232
WidgetNode::Tabs(_w) => &[],  // Tab 内容不可达
```

这意味着 `FocusManager::build_parent_map()` 和 `collect_focusable()` 不会进入 tab 内容区域。Tab 内的 Input/Button/List 永远不会被加入 tab_order，无法通过 Tab 键导航到。

**影响**：Tab 内容区内的可获焦组件不可达。

**修复方向**：`children()` 应返回当前活跃 tab 的内容节点切片。

## 三、中等偏差（功能缺失或集成断裂）

### 3.1 SIGWINCH 检测未集成到事件循环（TEP-0003）✅ 已修复

**TEP 设计**：终端尺寸变化 → 全树重排（SIGWINCH → full relayout）。

**代码现状**：`signal_manager.rs:49-58` 有 `check_resize()` 函数，但 `event_loop.rs:run_event_loop()` 从未调用它。事件循环只做 `poll_events(input)` 然后直接渲染。

```rust
// signal_manager.rs — 已实现但未调用
pub fn check_resize(app: &mut App, backend: &dyn TerminalBackend) -> bool {
    let (cols, rows) = backend.size();
    let (cur_cols, cur_rows) = app.screen_size();
    if cols != cur_cols || rows != cur_rows {
        app.resize(cols, rows);
        true
    } else {
        false
    }
}
```

Windows 上 resize 事件本来就不会通过 stdin 到达（crossterm 的 Event::Resize 只来自 SIGWINCH），但即便在 Unix 上也没有轮询接入。

**影响**：终端窗口缩放后渲染错位或截断，直到下次全量重绘才恢复。

**修复方向**：在 `run_event_loop` 起始或每轮循环中调用 `check_resize()`，检测到变化时 `app.mark_all_dirty()`。

---

### 3.2 缺少端到端测试（TEP-0006）✅ 已修复

**TEP 设计**：三层测试金字塔——domain 纯函数单元测试 → infra 模拟后端集成测试 → 端到端（模拟输入到后端输出断言）。

**原先缺失**：`pipeline_integration.rs` 覆盖了 measure→layout→render→diff→emit 管线，但只用 `ReadSignal::constant`，不涉及 Signal 变更 → DirtyTracker → 渲染的响应式闭环。

**修复**：新增 3 个 E2E 测试：
- `signal_set_marks_subscriber_dirty_and_renders_new_value` — 验证 Signal.set → mount 订阅 → dirty → 渲染输出新值
- `signal_same_value_does_not_mark_dirty` — 验证 PartialEq 跳过无变化更新
- `multiple_subscribers_all_marked_dirty` — 验证一个 Signal 驱动多个组件

**影响**：事件冒泡、焦点导航、Signal 响应式链路未被自动化验证。

---

### 3.3 arbor-tui crate 硬依赖 crossterm（TEP-0006）✅ 已修复

**TEP 设计**：三层依赖单向——app → infra → domain。infra 层封装所有 I/O 适配器。

**代码现状**：`arbor-tui/Cargo.toml` 不再依赖 crossterm。`arbor-tui/src/lib.rs` 的 re-export 通过 `arbor-tui-backend` 间接可用，但不强制引入 crossterm。

**修复**：
- 移除 `arbor-tui` 对 `crossterm` 的直接依赖
- 新建 `crates/arbor-tui-examples/` 独立 crate，作为 example 的唯一归宿，显式依赖 `crossterm`
- workspace 新增 `arbor-tui-examples` 成员
- 删除旧的 `crates/arbor-tui/examples/` 目录
- 依赖链变为：`arbor-tui-examples` → `crossterm` + `arbor-tui` → `arbor-tui-backend`（optional crossterm feature）→ `arbor-tui-core`

---

### 3.4 无基准测试验证硬约束（TEP-0001）✅ 已修复

**TEP 设计**：启动 < 50ms。1080p 全屏帧 < 5ms。空闲 CPU 接近零。内存 < 5MB。二进制 ≤ 8MB 静态链接。

**代码现状**：Release profile 配置了体积优化（opt-level="z", lto, strip, panic="abort"）。

**修复**：`app.rs` 新增 `FrameStats` 结构体和 `App::last_frame_stats` 字段。`render_widget_tree()` 在每次渲染时自动记录各阶段微秒级耗时（layout_us / render_us / diff_us / emit_us / total_us）和脏组件/脏区数量。调用方可直接读取 `app.last_frame_stats` 做性能观测和回归检测，无需 `profile` feature 或外部工具。

## 四、轻微偏差（细节不一致）

### 4.1 bind_signal 的 DirtyTracker 通知缺失（TEP-0004）

**代码现状**：`signal.rs:150-165` 的 `bind_signal()` 函数在 write closure 中直接修改 inner value，但不过 DirtyTracker。注释写明了这点：

```rust
// signal.rs:160-161
// Note: DirtyTracker notification happens via the generation bump.
// The caller should drain generations in the event loop.
```

"drain generations" 的逻辑在任何地方都没有实现。`bind_signal` 返回的 write closure 无法注入 DirtyTracker。

**影响**：低。bind_signal 是一个便利函数，不走它的话直接用 `signal.set(value, &mut dirty_tracker)` 是正确的。

---

### 4.2 Attrs 比 TEP 描述多 dim 和 reverse（TEP-0002）

**TEP 设计**：Cell 属性为"粗体/斜体/下划线等标记"。

**代码现状**：`Attrs` 包含 5 个 bool：bold, dim, italic, underline, reverse。TEP 只提了粗体/斜体/下划线，实际多了 dim 和 reverse。

**影响**：无。这是增强而非偏差。

---

### 4.3 弹性余数分配策略与 CSS 不一致（TEP-0003）

**TEP 设计**："弹性余数轮询分配"。

**代码现状**：`layout_engine.rs:353-369` 按 flex 降序排列 flex 子项，然后轮询分配余数像素。这不同于 CSS Flexbox 的"按 flex 比例分配"策略——轮询确保了整数像素分配确定性但可能不按比例。

**影响**：低。TEP 明确说这是手写 Flexbox 子集，整数像素分配策略是设计选择。1px 精度在终端网格中不可见。

---

### 4.4 Tab header 布局未使用布局引擎（TEP-0003 / TEP-0005）

**代码现状**：`render.rs:render_tabs()` 手动计算 tab header 位置——逐个拼接 label，没有走 measure/layout 两趟协商。`layout_engine.rs:layout_node()` 对 Tabs 变体硬编码了 header=1 行 + separator=1 行：

```rust
// layout_engine.rs:239-245
let header_rect = Rect::new(content_rect.x, content_rect.y, content_rect.w, 1);
let body_rect = Rect::new(
    content_rect.x, content_rect.y + 1,  // header 1 行，没有 separator
    content_rect.w, sat_sub(content_rect.h, 1),
);
```

注意：布局层给了 1 行 header，但渲染层实际画了 1 行 header + 1 行 separator + body = 至少 3 行。布局和渲染对 tab 区域高度的认知不一致。

**影响**：如果 rect.h < 3，body 区域高度可能溢出。

---

### 4.5 measure_tabs 的 header_h 未加入最终高度（TEP-0003）

**代码现状**：`layout_engine.rs:measure_tabs()` 定义 `header_h: u16 = 1` 然后 `min_h = header_h + max_h`，但没有算 separator 行。而渲染层 header + separator = 2 行。

**四条协议的整体评价**：

- **渲染协议（TEP-0002）**：实现度最高，Cell / VirtualScreen / diff / 后端适配器均按设计完成。无显著偏差。
- **布局协议（TEP-0003）**：两趟协商和 Flexbox 子集正确。SIGWINCH 集成缺失，Tab 布局渲染行数不一致。
- **响应式协议（TEP-0004）**：Signal/ReadSignal 机制正确，但自动订阅缺失和管线无条件执行导致实际响应式行为不工作。
- **组件协议（TEP-0005）**：WidgetNode 枚举和焦点管理正确。Tabs 有 2 个 bug，事件冒泡和生命周期调度框架就绪但不完整。
